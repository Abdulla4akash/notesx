<summary><strong>Q1.</strong> How do you recognize a BDI-style Gwendolen program?</summary>

Use: identify three layers.<br>1. Beliefs = facts currently in the belief base.<br>2. Goals = desired states or procedures the agent wants to pursue.<br>3. Intentions = active stacks of work generated by goals/events.<br>4. Plans connect events + current conditions to deeds.<br>Reference: BDI programming represents an agent using Beliefs, Desires/Goals, and Intentions.

</details>

<details>
<summary><strong>Q2.</strong> What is Gwendolen used for in this lecture cluster?</summary>

Use: treat it as the course’s BDI agent-programming language.<br>1. Read programs in terms of beliefs, goals, intentions, plans, events, and guards.<br>2. Remember it is chosen partly because it supports verification of agent programs.<br>3. Do not confuse it with lower-level robot control code.<br>Reference: Gwendolen is a BDI programming language used for high-level agent programming and later verification.

</details>

Gwendolen is used here because it is tied to the lecturer/course and verification material.<br>3. Core BDI concepts transfer across both.<br>Reference: both are BDI agent languages; Gwendolen is emphasized in this course.

</details>

<details>
<summary><strong>Q4.</strong> What does it mean that Gwendolen is high-level?</summary>

Use: separate agent decision logic from physical implementation.<br>1. A Gwendolen plan may contain an action deed like action(X).<br>2. The program does not specify motors, sensors, controllers, or execution physics.<br>3. Those details are delegated to the environment/low-level system.<br>Reference: high-level Gwendolen code specifies what the agent chooses to do, not how hardware carries it out.

</details>

<details>
<summary><strong>Q5.</strong> How do you recognize an action deed?</summary>

Use: inspect the deed syntax.<br>1. If the deed has no leading +, -, !, *, +.lock, or -.lock marker, treat it as an action call.<br>2. It asks the environment/low-level system to do something.<br>3. Do not treat it as directly adding/removing a belief unless it is written as +p or -p.<br>Reference: an action is a deed in a plan with no belief/goal/wait/lock decoration.

</details>

<details>
<summary><strong>Q6.</strong> Action deed vs belief-update deed — what is the discriminator?</summary>

Use this discriminator: does the deed directly change the belief base?<br>1. +p directly adds belief p.<br>2. -p directly removes belief p.<br>3. action(args) only requests an external action; any resulting beliefs arrive later via perception/events.<br>Reference: belief-update deeds modify beliefs; action deeds call the environment.

</details>

<details>
<summary><strong>Q7.</strong> What does it mean that BDI/Gwendolen is declarative?</summary>

Use: focus on what conditions and goals mean, not every control detail.<br>1. Plans are written around beliefs, goals, and guards.<br>2. The programmer states when a plan is applicable and what deeds follow.<br>3. The reasoning cycle handles plan selection/execution machinery.<br>Reference: declarative agent programming expresses problem structure using beliefs, goals, and conditions.

</details>

<details>
<summary><strong>Q8.</strong> Declarative BDI caveat — what should you say in an exam?</summary>

Use: avoid claiming Gwendolen programs are always simple.<br>1. BDI languages aim to be declarative.<br>2. Real programs may still need many plans for the same event.<br>3. Different guards handle different belief/configuration cases.<br>Reference: non-trivial BDI programs can still contain complicated guards and many case-split plans.

</details>

Read :Plans: for event-triggered behaviour.<br>Reference: a Gwendolen program is organized into name, initial beliefs, initial goals, and plans.

</details>

<details>
<summary><strong>Q10.</strong> What does the :name: section do?</summary>

Use: identify which agent the program belongs to.<br>1. Read :name: as the agent identifier.<br>2. This matters when there may be multiple agents.<br>Reference: :name: names the agent.

</details>

<details>
<summary><strong>Q11.</strong> What does :Initial Beliefs: do?</summary>

Use: initialize the belief base.<br>1. Each listed fact is initially believed by the agent.<br>2. These beliefs can satisfy plan guards from the start.<br>3. They may later be added to or removed by deeds/perception.<br>Reference: :Initial Beliefs: lists facts initially in the agent’s belief base.

</details>

<details>
<summary><strong>Q12.</strong> What does :Initial Goals: do?</summary>

Use: identify what the agent starts trying to do.<br>1. Each listed goal is inserted via the start-event machinery.<br>2. Mark whether each goal is [achieve] or [perform].<br>3. Apply the corresponding goal-handling procedure.<br>Reference: :Initial Goals: lists goals initially given to the agent.

</details>

<details>
<summary><strong>Q13.</strong> What does :Plans: contain?</summary>

Use: read each plan as event + condition + response.<br>1. Trigger: what event has occurred.<br>2. Guard: what must currently hold.<br>3. Deeds: what the agent should execute if the plan applies.<br>Reference: :Plans: contains rules of form trigger : {guard} <- deed1, ..., deedN;.

</details>

<details>
<summary><strong>Q14.</strong> How do you parse a plan of the form trigger : {guard} <- deed1, ..., deedN;?</summary>

Use this method.<br>1. Left of ':' = trigger event.<br>2. Inside '{...}' = guard conditions checked now.<br>3. Right of '<-' = ordered deed sequence.<br>4. The plan applies only when the trigger occurs and the guard is satisfied.<br>Reference: a plan maps an event under a guard to a sequence of deeds.

</details>

<details>
<summary><strong>Q15.</strong> How do you parse trigger syntax with +, -, and !?</summary>

Use the symbols first.<br>1. +p = belief p has appeared/been added.<br>2. -p = belief p has disappeared/been removed.<br>3. +!g [achieve] = achievement goal g has appeared.<br>4. -!g [achieve] = achievement goal g has been removed.<br>5. ! means goal; no ! means belief.<br>Reference: triggers are belief/goal addition or removal events.

</details>

<details>
<summary><strong>Q16.</strong> How do you read common guard literals?</summary>

Use this lookup.<br>1. B p = agent believes p.<br>2. ~B p = agent does not believe p.<br>3. G g [achieve] = agent currently has achievement goal g.<br>4. True or T = guard always succeeds.<br>Reference: a guard is a current-condition test inside {...}.

</details>

<details>
<summary><strong>Q17.</strong> How do you classify a deed in a Gwendolen plan?</summary>

Use the first marker.<br>1. +p / -p = add/remove belief.<br>2. +!g / -!g = add/remove goal.<br>3. action(args) = call external action.<br>4. *p = wait for belief p.<br>5. +.lock / -.lock = start/end locked segment.<br>Reference: deeds are the executable items on the right-hand side of a plan.

</details>

<details>
<summary><strong>Q18.</strong> What does an always-true guard mean?</summary>

Use: if the guard is {T} or {True}, do not look for extra belief conditions.<br>1. The plan is applicable whenever its trigger occurs.<br>2. Variables still need to be bound by the trigger or other context if used in deeds.<br>Reference: T/True in a guard means the guard condition always holds.

</details>

<details>
<summary><strong>Q19.</strong> How do you execute a perform goal?</summary>

Use this procedure.<br>1. Add the perform goal as an event.<br>2. Find an applicable plan for that goal.<br>3. Execute that plan’s deeds.<br>4. Drop the goal; do not re-check whether a belief became true.<br>Reference: a perform goal means “do this procedure,” like a subroutine.

</details>

<details>
<summary><strong>Q20.</strong> How do you execute an achieve goal?</summary>

Use this loop.<br>1. Check whether the corresponding belief g is already in the belief base.<br>2. If yes, the goal is already achieved.<br>3. If no, find and execute an applicable plan for +!g [achieve].<br>4. After the plan, check the belief base again.<br>5. If g is still not believed, plan again.<br>Reference: an achieve goal means “make g true and keep trying until g is believed.”

</details>

<details>
<summary><strong>Q21.</strong> Achieve goal vs perform goal — what is the discriminator?</summary>

Discriminator: is success judged by a resulting belief?<br>1. [achieve]: yes; keep trying until the corresponding belief is believed.<br>2. [perform]: no; execute the procedure and then drop it.<br>Reference: achieve goals are state-oriented; perform goals are procedure-oriented.

</details>

<details>
<summary><strong>Q22.</strong> What happens if an achieve goal is already believed?</summary>

Use: check before planning.<br>1. When +!g [achieve] is processed, inspect the belief base for g.<br>2. If g is already believed, no achievement plan is needed.<br>3. If g is absent, plan for the achievement event.<br>Reference: achieve goals are checked against the belief base before plan execution.

</details>

<details>
<summary><strong>Q23.</strong> What happens if an achieve goal is still not believed after a plan?</summary>

Use: do not drop it prematurely.<br>1. Re-check whether g is in the belief base.<br>2. If not, the achievement goal remains active.<br>3. Gwendolen may select another applicable plan for +!g [achieve].<br>Reference: achieve goals persist until the corresponding belief is believed.

</details>

<details>
<summary><strong>Q24.</strong> How are initial goals inserted into execution?</summary>

Use: model them through the start event.<br>1. Initial goal g becomes a start-triggered deed +!g.<br>2. Processing that deed creates a goal event.<br>3. That event is then planned/executed according to [achieve] or [perform].<br>Reference: initial goals use the special start event to enter the agent machinery.

</details>

<details>
<summary><strong>Q25.</strong> What is the Gwendolen reasoning cycle?</summary>

Use this stage order.<br>1. Select a current intention, or sleep/go to perception if none is usable.<br>2. If needed, find applicable plans for the current intention.<br>3. Pick/apply a plan, adding deeds to the stack.<br>4. Execute the top deed.<br>5. Get perceptions and messages.<br>6. Process messages, then return to intention selection.<br>Reference: Gwendolen repeatedly selects intentions, plans, executes deeds, then handles perception/messages.

</details>

<details>
<summary><strong>Q26.</strong> When does Gwendolen search for applicable plans?</summary>

Use: inspect the top deed.<br>1. If the current intention’s top deed is no plan yet, planning has not happened.<br>2. Gwendolen matches plans whose trigger fits the event.<br>3. It keeps only plans whose guards are satisfied.<br>4. It selects one applicable plan and pushes its deeds onto the intention.<br>Reference: planning occurs when an event on an intention needs a plan.

</details>

<details>
<summary><strong>Q27.</strong> What happens when multiple plans apply?</summary>

Use: do not assume all applicable plans execute.<br>1. Gwendolen finds applicable plans for the current event/guard.<br>2. If more than one applies, one plan is selected.<br>3. The selected plan’s deeds are added to the current intention.<br>Reference: an applicable plan is a candidate response; execution follows the chosen plan.

</details>

<details>
<summary><strong>Q28.</strong> Why does the order of the reasoning cycle matter?</summary>

Use: track perception lag.<br>1. An action may cause new perceptions.<br>2. Perceptions do not instantly become beliefs.<br>3. They create intentions that must later be selected and processed.<br>4. Until then, guards/achieve checks cannot rely on those beliefs.<br>Reference: action effects reach the belief base only through later perception-processing intentions.

</details>

<details>
<summary><strong>Q29.</strong> How do you explain a repeated action caused by perception lag?</summary>

Use this causal chain.<br>1. An action makes condition p true in the environment.<br>2. Perception of p is queued as an intention, not instantly added as belief p.<br>3. Before that intention runs, an achieve goal checks for p and finds it absent.<br>4. The achievement plan may run again.<br>Reference: perception lag can make an achieve goal retry before the confirming belief is added.

</details>

<details>
<summary><strong>Q30.</strong> What happens when there are no usable non-empty intentions?</summary>

Use: follow the cycle branch.<br>1. If no current intention can proceed, Gwendolen may sleep or move to perception/message handling.<br>2. New perceptions/messages may create fresh intentions.<br>Reference: the cycle selects an intention if possible; otherwise it waits for new work via perception/messages.

</details>

<details>
<summary><strong>Q31.</strong> What is an event in Gwendolen?</summary>

Use: look for a change, not a static fact.<br>1. A belief being added is an event: +p.<br>2. A belief being removed is an event: -p.<br>3. A goal being added/removed is an event: +!g or -!g.<br>4. Plans fire on events plus guards.<br>Reference: an event is a belief or goal addition/removal.

</details>

<details>
<summary><strong>Q32.</strong> Event trigger vs condition — what is the discriminator?</summary>

Discriminator: did something just change?<br>1. Trigger = the change that just occurred, such as +p.<br>2. Guard = a condition that currently holds, such as B q or ~B q.<br>3. A plan does not re-fire merely because p remains true.<br>Reference: triggers are events; guards are current conditions.

</details>

<details>
<summary><strong>Q33.</strong> Why does a plan not repeatedly fire while a belief remains true?</summary>

Use: distinguish persistence from appearance.<br>1. A trigger +p occurs only when p is added/appears.<br>2. If p stays in the belief base, no new +p event happens.<br>3. The plan fires again only if p is removed and later re-added, or another matching event occurs.<br>Reference: plans are event-triggered, not continuously condition-triggered.

</details>

<details>
<summary><strong>Q34.</strong> What is the special start event for?</summary>

Use: identify inserted initial/perceptual work.<br>1. Initial goals are inserted using start.<br>2. Perceptual information to be added as beliefs is also inserted using start.<br>3. The start event creates an intention whose deed performs the actual belief/goal update.<br>Reference: start is a special event used to introduce initial goals and perceptual beliefs.

</details>

<details>
<summary><strong>Q35.</strong> How does perception become a belief?</summary>

Use: do not add it instantly.<br>1. The environment supplies a percept p.<br>2. Gwendolen creates an intention, typically with start and deed +p.<br>3. Only when that intention executes +p does p enter the belief base.<br>Reference: perception creates intentions; beliefs change when those intentions are processed.

</details>

<details>
<summary><strong>Q36.</strong> What is an intention in Gwendolen?</summary>

Use: treat it as a stack of execution tuples.<br>1. Each tuple records an event/deed context.<br>2. The top tuple determines what the intention does next.<br>3. Plans add new deeds on top of the stack.<br>Reference: an intention is a stack of tuples of the form (trigger event, deed, unifier).

</details>

<details>
<summary><strong>Q37.</strong> What are the three components of an intention tuple?</summary>

Use this tuple schema.<br>1. Trigger event = the event being handled.<br>2. Deed = the current instruction to execute.<br>3. Unifier = variable bindings produced by matching.<br>Reference: intention tuple = (trigger event, deed, unifier).

</details>

<details>
<summary><strong>Q38.</strong> What is a unifier used for?</summary>

Use: track variable bindings.<br>1. Match the trigger/guard against current events and beliefs.<br>2. Bind variables to values or terms.<br>3. Carry those bindings into the selected plan’s deeds.<br>Reference: a unifier records variable bindings produced during plan matching.

</details>

<details>
<summary><strong>Q39.</strong> What is a deed?</summary>

Use: look at the executable item in the intention tuple.<br>1. It may be an action, belief update, goal update, wait, lock/unlock, no plan yet, or null.<br>2. The reasoning cycle executes or handles the top deed.<br>Reference: a deed is the current instruction stored in an intention tuple.

</details>

<details>
<summary><strong>Q40.</strong> What does no plan yet mean?</summary>

Use: interpret it as a planning placeholder.<br>1. The intention contains an event.<br>2. Gwendolen has not yet selected a plan for that event.<br>3. When this is on top, Gwendolen searches for applicable plans.<br>Reference: no plan yet means the event exists but planning has not happened.

</details>

<details>
<summary><strong>Q41.</strong> What does npy stand for in intention-stack slides?</summary>

Use: normalize the abbreviation.<br>1. Read npy as no plan yet.<br>2. It means the event still needs plan selection.<br>Reference: npy/no plan yet is the special deed indicating planning is pending.

</details>

<details>
<summary><strong>Q42.</strong> What does null mean in a perform-goal intention?</summary>

Use: distinguish it from an action.<br>1. The original perform-goal deed is replaced by null when it becomes a new event.<br>2. The perform goal is planned/executed through the new event.<br>3. The old perform-goal deed is not kept for achievement checking.<br>Reference: null marks the consumed perform-goal deed; perform goals disappear after their plan executes.

</details>

<details>
<summary><strong>Q43.</strong> How do you apply a plan to an intention stack?</summary>

Use this method.<br>1. Match the current event to a plan trigger.<br>2. Check the guard under a unifier.<br>3. Remove/replace no plan yet.<br>4. Push the selected plan’s deeds onto the current intention in execution order.<br>Reference: plan application expands an event into executable deeds on the intention stack.

</details>

<details>
<summary><strong>Q44.</strong> How do perform goals differ from achieve goals in the intention stack?</summary>

Use: inspect what remains after goal expansion.<br>1. Perform goal: original deed becomes null; once planned/executed, it disappears.<br>2. Achieve goal: goal remains available for later achievement checking.<br>Reference: perform goals are not maintained for belief-based success; achieve goals are.

</details>

<details>
<summary><strong>Q45.</strong> What starts a new intention?</summary>

Use: look for a new event to react to.<br>1. Initial goal insertion can start an intention.<br>2. Perception becoming a belief can start an intention.<br>3. Adding/removing beliefs can start intentions.<br>4. Adding/removing goals can start intentions.<br>Reference: each event being reacted to can start an intention.

</details>

<details>
<summary><strong>Q46.</strong> Why are multiple intentions useful?</summary>

Use: explain responsiveness.<br>1. The agent can pursue an existing goal.<br>2. New events/percepts can create separate intentions.<br>3. The agent can react to new information without abandoning the original goal.<br>Reference: multiple intentions allow interleaved pursuit of goals and reactions to events.

</details>

<details>
<summary><strong>Q47.</strong> Why can multiple intentions cause bugs?</summary>

Use: look for harmful interleaving.<br>1. Gwendolen may switch away from a partially completed sequence.<br>2. Another intention may observe the belief base in an intermediate state.<br>3. The original intention may continue before related perception has been processed.<br>Reference: interleaving improves responsiveness but can expose half-finished updates or stale beliefs.

</details>

<details>
<summary><strong>Q48.</strong> How do you diagnose an interleaving bug?</summary>

Use this checklist.<br>1. Identify the sequence that was meant to happen together.<br>2. Mark each deed where the agent could switch intentions.<br>3. Ask whether another intention can see missing/stale/intermediate beliefs.<br>4. If yes, consider waiting or locking.<br>Reference: interleaving bugs arise when intention switching changes what guards/goals see.

</details>

<details>
<summary><strong>Q49.</strong> What is a plan guard?</summary>

Use: treat it as an applicability test.<br>1. It is checked only when the triggering event is being handled.<br>2. It queries current beliefs/goals/reasoning rules.<br>3. If it fails, that plan is not applicable for the event.<br>Reference: a guard is the condition inside {...} that must hold for a plan to apply.

</details>

<details>
<summary><strong>Q50.</strong> Guard vs trigger — how do you decide where a fact belongs?</summary>

Discriminator: is it a change or a required current state?<br>1. Put the event/change before ':' as the trigger.<br>2. Put required beliefs/goals inside the guard.<br>3. Do not use a trigger for a fact that merely needs to remain true.<br>Reference: triggers handle events; guards test current conditions.

</details>

<details>
<summary><strong>Q51.</strong> How do you evaluate a guard with variables?</summary>

Use this method.<br>1. Start from variable bindings supplied by the trigger.<br>2. Search for bindings that make all guard literals true.<br>3. Apply the resulting unifier to the plan deeds.<br>4. If no binding satisfies the guard, the plan is not applicable.<br>Reference: guard evaluation uses unification against current beliefs/goals/derived facts.

</details>

<details>
<summary><strong>Q52.</strong> What is a reasoning rule in Gwendolen?</summary>

Use: treat it as a derived belief rule for guards.<br>1. Define a head predicate from body conditions.<br>2. Use the derived head in a plan guard.<br>3. This keeps the plan guard simpler.<br>Reference: reasoning rules have Prolog/Datalog-like form Head :- Body;.

</details>

<details>
<summary><strong>Q53.</strong> How do you read Head :- Body;?</summary>

Use: read 'if' from right to left.<br>1. Check whether every condition in Body holds.<br>2. If Body holds under some binding, derive Head under that binding.<br>3. The derived Head can satisfy B Head in a guard.<br>Reference: Head :- Body means Head holds if Body holds.

</details>

<details>
<summary><strong>Q54.</strong> When should you factor a guard into a reasoning rule?</summary>

Use: when guard logic is reusable or cluttered.<br>1. Move multi-condition logic into a named derived predicate d(X).<br>2. Define d(X) with reasoning rules.<br>3. Replace the long guard with B d(X).<br>Reference: reasoning rules simplify guards by naming derived conditions.

</details>

<details>
<summary><strong>Q55.</strong> How do you use a reasoning-rule predicate inside a plan guard?</summary>

Use this pattern.<br>1. Define d(X) :- condition1(X), condition2(X);.<br>2. In the plan, write {B d(X)}.<br>3. Gwendolen derives possible X values through the reasoning rule.<br>4. The unifier from d(X) is used in the deeds.<br>Reference: derived predicates can be queried as believed/derivable guard conditions.

</details>

<details>
<summary><strong>Q56.</strong> What does negation in a reasoning-rule body do?</summary>

Use: treat ~q(X) as a negative condition on derivability/belief.<br>1. Try to prove or find q(X).<br>2. If q(X) is not present/derivable for that binding, ~q(X) succeeds.<br>3. If q(X) is present/derivable, ~q(X) fails.<br>Reference: ~ in rule bodies expresses a not-known/not-derivable condition.

</details>

<details>
<summary><strong>Q57.</strong> How do reasoning rules connect to Datalog/Prolog?</summary>

Use: mention rule syntax and unification.<br>1. Rules use Head :- Body form.<br>2. Variables are bound by unification.<br>3. Recursive rules can process structured terms such as lists.<br>Reference: Gwendolen reasoning rules are Prolog/Datalog-like in style.

</details>

<details>
<summary><strong>Q58.</strong> How do you read Prolog-style list notation [Head | Tail]?</summary>

Use: split the list into first element and remainder.<br>1. Head matches the first item.<br>2. Tail matches the rest of the list.<br>3. Use the binding in recursive rules.<br>Reference: [Head | Tail] decomposes a non-empty list into its head and tail.

</details>

<details>
<summary><strong>Q59.</strong> How do you write a recursive rule that selects the first usable list item?</summary>

Use two rules.<br>1. Base case: select([Item | Tail], Item) succeeds if Item is usable.<br>2. Recursive case: select([Item | Tail], X) recurses on Tail if Item is not usable or already handled.<br>3. The first rule that succeeds returns the selected item.<br>Reference: recursive list rules combine a head-of-list case with a tail-recursion case.

</details>

<details>
<summary><strong>Q60.</strong> How do you trace a recursive list guard?</summary>

Use this method.<br>1. Match the list against [Head | Tail].<br>2. Test the base-case condition for Head.<br>3. If it succeeds, return Head’s binding.<br>4. If it fails and the recursive condition holds, recurse on Tail.<br>5. Repeat until a base case succeeds or the list is exhausted.<br>Reference: recursive guards derive values by repeatedly decomposing a list.

</details>

<details>
<summary><strong>Q61.</strong> What does a base case do in a list-processing reasoning rule?</summary>

Use: check whether the current head already satisfies the target condition.<br>1. Match [Head | Tail].<br>2. Test the condition for Head.<br>3. If it holds, derive the target predicate from Head.<br>Reference: the base case stops recursion when the current list item is usable.

</details>

<details>
<summary><strong>Q62.</strong> What does the recursive case do in a list-processing reasoning rule?</summary>

Use: skip an unusable/handled head.<br>1. Match [Head | Tail].<br>2. Confirm Head should be skipped.<br>3. Re-run the same predicate on Tail.<br>Reference: the recursive case moves from the list head to the tail.

</details>

<details>
<summary><strong>Q63.</strong> Can goals appear in guards?</summary>

Use: yes, with G-goal notation.<br>1. Write G g [achieve] to test whether the agent currently has achievement goal g.<br>2. Combine it with belief guards when an action should depend on the current goal.<br>Reference: guards can test current goals as well as beliefs.

</details>

<details>
<summary><strong>Q64.</strong> Why add a goal condition to a guard?</summary>

Use: restrict a reaction to relevant goals.<br>1. A percept/event may occur in many contexts.<br>2. Add G g [achieve] when the plan should run only while pursuing g.<br>3. This prevents the agent reacting to the event when it is irrelevant to its current objective.<br>Reference: a goal guard checks whether the agent currently has that goal.

</details>

<details>
<summary><strong>Q65.</strong> How do you read a guarded reaction of the form +p(X) : {B q(X), G g [achieve]} <- action(X);?</summary>

Use this parse.<br>1. Trigger: p(X) has appeared.<br>2. Belief condition: q(X) must currently be believed.<br>3. Goal condition: g [achieve] must currently be an active goal.<br>4. If both hold, execute action(X).<br>Reference: event-triggered plans can be gated by both belief and goal guards.

</details>

<details>
<summary><strong>Q66.</strong> How do you model a higher-level achievement goal that needs a subgoal first?</summary>

Use this plan pattern.<br>1. For +!g_final [achieve], adopt +!g_sub [achieve].<br>2. After the subgoal deed, place follow-up actions/deeds needed for g_final.<br>3. Guard low-level reactions with G g_sub [achieve] if they should happen only during the subgoal.<br>Reference: achievement plans may create subgoals as deeds before continuing.

</details>

<details>
<summary><strong>Q67.</strong> What problem does waiting solve?</summary>

Use: look for an intention continuing too early.<br>1. An action creates perceptions to be processed by other intentions.<br>2. The original intention may otherwise continue before those perceptions update beliefs.<br>3. Insert a wait deed for a marker belief that says processing is complete.<br>Reference: waiting prevents premature continuation of one intention while related events are still being handled.

</details>

<details>
<summary><strong>Q68.</strong> What is the syntax of a wait deed?</summary>

Use: look for the star.<br>1. *p means wait until p is believed.<br>2. It is a deed inside a plan sequence.<br>3. It does not itself add p.<br>Reference: *p means the current intention should not proceed until the agent believes p.

</details>

<details>
<summary><strong>Q69.</strong> How does Gwendolen handle an unsatisfied wait deed?</summary>

<summary><strong>Q73.</strong> What exactly does waiting fix in a multi-intention program?</summary>

Use: name the race condition.<br>1. Without waiting, an achievement intention may race ahead before current percepts are processed.<br>2. With waiting, that intention stops at *p until another intention records completion by adding p.<br>3. Responsiveness is preserved because other intentions still run.<br>Reference: waiting controls progress of one intention while allowing interleaving elsewhere.

<summary><strong>Q74.</strong> What problem does locking solve?</summary>

Use: look for a short sequence that must be atomic.<br>1. Some belief/configuration updates must happen together.<br>2. If another intention runs halfway through, it may see an inconsistent intermediate state.<br>3. Put a lock around the short critical update.<br>Reference: locking prevents interleaving during a locked segment of one intention.

</details>

<summary><strong>Q75.</strong> How do mode beliefs control plans?</summary>

<summary><strong>Q77.</strong> What is the syntax of locking?</summary>

Use lock/unlock deeds.<br>1. +.lock starts a locked segment.<br>2. Place the short critical update deeds next.<br>3. -.lock releases the lock.<br>Reference: between +.lock and -.lock, the current intention remains selected.

<summary><strong>Q78.</strong> How do you trace a locked update?</summary>

Use this method.<br>1. Execute +.lock.<br>2. Execute each internal update deed in order.<br>3. Do not switch to other intentions during the locked segment.<br>4. Execute -.lock.<br>5. After unlock, normal intention interleaving resumes.<br>Reference: locking keeps the current intention current until released.

</details>

<summary><strong>Q79.</strong> What is a good use of lock?</summary>

Use: only lock short internal updates.<br>1. Lock belief/configuration updates that must not be observed halfway through.<br>2. Keep the locked section small.<br>3. <summary><strong>Q80.</strong> What is a bad use of lock?</summary>

<summary><strong>Q81.</strong> Why does locking reduce responsiveness?</summary>

This is safe for short atomic updates but risky for long actions.<br>Reference: locking prevents interleaving by temporarily blocking other intentions.
