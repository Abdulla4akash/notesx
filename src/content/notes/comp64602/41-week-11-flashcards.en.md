# Week 11 — Flashcards

<details>
<summary><strong>Q1.</strong> Discriminator: verification vs validation — which question are you answering?</summary>

Use this test first:<br>1. If the question is whether the implementation satisfies a formal specification, call it <b>verification</b>.<br>2. If the question is whether the chosen requirements/system match the real-world need, call it <b>validation</b>.<br>Reference: verification = “did we build the system right?”; validation = “did we build the right system?”

</details>

<details>
<summary><strong>Q2.</strong> Discriminator: testing vs model checking — selected executions or all generated executions?</summary>

Use this discriminator:<br>1. If only chosen/sample runs are checked, it is <b>testing</b>.<br>2. If every execution generated by the model is checked against a property, it is <b>model checking</b>.<br>Reference: testing is path/sample based; model checking explores the transition system exhaustively.

</details>

<details>
<summary><strong>Q3.</strong> Discriminator: model checking vs theorem proving — exploration or symbolic proof?</summary>

Use this discriminator:<br>1. If the method explores model executions/branches, classify it as <b>model checking</b>.<br>2. If the method derives the property symbolically from formulas/rules, classify it as <b>theorem proving</b>.<br>Reference: model checking searches a transition model; theorem proving reasons symbolically.

</details>

<details>
<summary><strong>Q4.</strong> When should a property be written in a formal specification language?</summary>

Use it when the claim must be checked precisely rather than informally.<br>Steps:<br>1. Identify the system behaviour to verify.<br>2. Express it in a logic whose operators match the behaviour over time.<br>3. Give the formula to the verification method.<br>Reference: temporal logics express how programs/systems change as execution unfolds.

</details>

<details>
<summary><strong>Q5.</strong> Build a transition-system model from a program or component.</summary>

Method:<br>1. Choose the state set S.<br>2. Define the transition relation → between states.<br>3. Choose the atomic propositions Π.<br>4. Define π so each proposition maps to the states where it is true.<br>5. Choose the initial state s₀.<br>Reference: transition system = ⟨S, →, Π, π, s₀⟩.

</details>

<details>
<summary><strong>Q6.</strong> Use the valuation function π to decide whether proposition p holds.</summary>

Method:<br>1. For a state-level check, ask whether the current state is in π(p).<br>2. For a sequence-position check, find the propositions true at position i.<br>3. Accept p exactly when p ∈ π(i).<br>Reference: π maps propositions to states; in a sequence, π(i) is the set of propositions true at position i.

</details>

<details>
<summary><strong>Q7.</strong> Run model checking once a transition system and formula φ are given.</summary>

Method:<br>1. Model the program/component as a transition system.<br>2. Generate the set Σ of possible execution sequences.<br>3. Check every σ ∈ Σ against φ from the initial state.<br>4. If every check succeeds, the model satisfies φ.<br>5. If one fails, return that failing sequence as a counter-example.<br>Reference: check ∀σ ∈ Σ . ⟨σ,s₀⟩ ⊨ φ.

</details>

<details>
<summary><strong>Q8.</strong> What is a counter-example in model checking?</summary>

Use it as the diagnostic output when verification fails.<br>Steps:<br>1. Find an execution sequence σ that violates φ.<br>2. Report σ as evidence that the property is not guaranteed.<br>3. Read it as “here is one possible run where the bad thing happens.”<br>Reference: a counter-example is a failing execution sequence.

</details>

<details>
<summary><strong>Q10.</strong> Check an atomic proposition at sequence position i.</summary>

Method:<br>1. Look up the set of propositions true at position i.<br>2. Check whether p is a member of that set.<br>3. If yes, the atomic proposition is satisfied; otherwise it is not.<br>Reference: ⟨σ,sᵢ⟩ ⊨ p iff p ∈ π(i).

</details>

<details>
<summary><strong>Q11.</strong> Check ¬φ at sequence position i.</summary>

Method:<br>1. First check whether φ holds at position i.<br>2. Flip the result.<br>3. ¬φ is true exactly when φ is not true.<br>Reference: ⟨σ,sᵢ⟩ ⊨ ¬φ iff not(⟨σ,sᵢ⟩ ⊨ φ).

</details>

<details>
<summary><strong>Q12.</strong> Check φ ∧ ψ at sequence position i.</summary>

Method:<br>1. Check φ at position i.<br>2. Check ψ at position i.<br>3. The conjunction succeeds only if both checks succeed.<br>Reference: ⟨σ,sᵢ⟩ ⊨ φ ∧ ψ iff ⟨σ,sᵢ⟩ ⊨ φ and ⟨σ,sᵢ⟩ ⊨ ψ.

</details>

<details>
<summary><strong>Q13.</strong> Check φ ∨ ψ at sequence position i.</summary>

Method:<br>1. Check φ at position i.<br>2. Check ψ at position i.<br>3. The disjunction succeeds if at least one check succeeds.<br>Reference: ⟨σ,sᵢ⟩ ⊨ φ ∨ ψ iff ⟨σ,sᵢ⟩ ⊨ φ or ⟨σ,sᵢ⟩ ⊨ ψ.

</details>

<details>
<summary><strong>Q14.</strong> Check ○φ, the next operator, at sequence position i.</summary>

Method:<br>1. Move one position forward, from i to i+1.<br>2. Check φ there.<br>3. If φ holds at i+1, then ○φ holds at i.<br>Reference: ⟨σ,sᵢ⟩ ⊨ ○φ iff ⟨σ,sᵢ₊₁⟩ ⊨ φ.

</details>

<details>
<summary><strong>Q15.</strong> Check ◇φ, the eventually operator, at sequence position i.</summary>

Method:<br>1. Search positions j from i onward.<br>2. Stop when you find a j where φ holds.<br>3. If such a j exists, ◇φ is true; if none exists, it is false.<br>Reference: ⟨σ,sᵢ⟩ ⊨ ◇φ iff ∃j ≥ i such that ⟨σ,sⱼ⟩ ⊨ φ.

</details>

<details>
<summary><strong>Q16.</strong> Check □φ, the always operator, at sequence position i.</summary>

Method:<br>1. Inspect every position j from i onward.<br>2. Check φ at each j.<br>3. If any future/current position violates φ, □φ is false.<br>Reference: ⟨σ,sᵢ⟩ ⊨ □φ iff for all j ≥ i, ⟨σ,sⱼ⟩ ⊨ φ.

</details>

<details>
<summary><strong>Q17.</strong> Check φ U ψ, the until operator, at sequence position i.</summary>

Method:<br>1. Find a future/current position j ≥ i where ψ holds.<br>2. Check every intermediate position k with i ≤ k &lt; j.<br>3. φ must hold at every such k.<br>4. If both conditions hold, φ U ψ is true.<br>Reference: ⟨σ,sᵢ⟩ ⊨ φ U ψ iff ψ eventually holds and φ holds until then.

</details>

<details>
<summary><strong>Q18.</strong> Reduce a conjunction of temporal claims to individual checks.</summary>

Method:<br>1. Split the formula at each top-level ∧.<br>2. Evaluate each conjunct using its own operator rule.<br>3. If any conjunct fails, the whole formula fails.<br>4. If all conjuncts pass, the whole formula passes.<br>Reference: conjunction is satisfied exactly when every conjunct is satisfied.

</details>

<details>
<summary><strong>Q19.</strong> Discriminator: LTL vs CTL/CTL* — all paths implicit or path quantifiers explicit?</summary>

Use this discriminator:<br>1. If the formula is interpreted over all model paths by default, think LTL-style.<br>2. If the formula explicitly says “all paths” or “some path” using A/E, think CTL/CTL*.<br>Reference: CTL* makes path quantification explicit with A and E.

</details>

<details>
<summary><strong>Q20.</strong> Use CTL* path quantifiers A and E.</summary>

Method:<br>1. Use Aφ when φ must hold on every path from the current state.<br>2. Use Eφ when φ must hold on at least one path from the current state.<br>3. Put a path quantifier before temporal reasoning about paths.<br>Reference: A = universal path quantifier; E = existential path quantifier.

</details>

<details>
<summary><strong>Q21.</strong> Translate an LTL formula φ into CTL* form.</summary>

Method:<br>1. Keep the temporal structure of φ.<br>2. Add the universal path quantifier A in front.<br>3. Read the result as “φ holds on all paths.”<br>Reference: any LTL formula φ translates into CTL* as Aφ.

</details>

<details>
<summary><strong>Q22.</strong> Discriminator: CTL vs CTL* — where must the path quantifier sit?</summary>

Use this test:<br>1. Find every temporal operator, such as ○, □, ◇, or U.<br>2. In CTL, each temporal operator must be immediately preceded by A or E.<br>3. In CTL*, a path quantifier only needs to occur somewhere before the temporal operator.<br>Reference: CTL is the immediate-path-quantifier fragment of CTL*.

</details>

<details>
<summary><strong>Q23.</strong> Decide whether a formula is valid CTL syntax.</summary>

Method:<br>1. Scan for temporal operators.<br>2. Check the symbol immediately before each temporal operator’s temporal subformula.<br>3. If each one is paired with A or E, classify it as CTL.<br>4. If a temporal operator is unpaired, classify it as CTL* but not CTL.<br>Reference: CTL requires every temporal operator to be immediately preceded by a path quantifier.

</details>

<details>
<summary><strong>Q24.</strong> CTL schema: “it is always possible for p to hold next.”</summary>

Method:<br>1. “Always” over all reachable states gives A□.<br>2. “Possible” gives E.<br>3. “Next p” gives ○p.<br>4. Combine as A□E○p.<br>Reference: A□E○p means: on all paths, always, there exists a path where p holds next.

</details>

<details>
<summary><strong>Q25.</strong> CTL schema: “whenever q holds, r is eventually reached on all paths.”</summary>

Method:<br>1. Use A□ to check every reachable point.<br>2. Use q ⇒ ... for the trigger.<br>3. Use A◇r for the required eventual outcome on all paths.<br>4. Combine as A□(q ⇒ A◇r).<br>Reference: A□(q ⇒ A◇r) captures universal response/liveness after q.

</details>

<details>
<summary><strong>Q26.</strong> CTL schema: “there exists a path where p is eventually reached.”</summary>

Method:<br>1. “There exists a path” gives E.<br>2. “Eventually p” gives ◇p.<br>3. Combine as E◇p.<br>Reference: E◇p means some path reaches a p-state eventually.

</details>

<details>
<summary><strong>Q27.</strong> CTL schema: “from every reachable state, p can eventually be reached.”</summary>

Method:<br>1. Use A□ for every reachable state along all paths.<br>2. Use E◇p for “there is some path to eventual p.”<br>3. Combine as A□E◇p.<br>Reference: A□E◇p means it is always possible to reach p eventually.

</details>

<summary><strong>Q31.</strong> Use the ATL cooperation modality ⟪C⟫φ.</summary>

Method:<br>1. Identify the coalition C.<br>2. Identify the outcome property φ.<br>3. Write ⟪C⟫φ to say C has the power/strategy to ensure φ.<br>Reference: ⟪C⟫φ means team C can ensure φ.

</details>

<details>
<summary><strong>Q32.</strong> Decide whether an ATL* formula is in the ATL fragment.</summary>

Method:<br>1. Find each temporal operator.<br>2. Check whether a cooperation modality immediately precedes it.<br>3. If yes for every temporal operator, it fits the ATL-style restriction.<br>4. If not, it is ATL* rather than ATL.<br>Reference: ATL mirrors the CTL restriction; the cooperation modality must immediately precede the temporal operator.

</details>

<details>
<summary><strong>Q33.</strong> ATL schema: “if q holds, coalition C can ensure p eventually.”</summary>

Method:<br>1. Put the condition first: q ⇒ ...<br>2. Use ⟪C⟫ for coalition ability.<br>3. Use ◇p for eventual p.<br>4. Combine as q ⇒ ⟪C⟫◇p.<br>Reference: ⟪C⟫◇p means coalition C has a strategy to make p eventually true.

</details>

<details>
<summary><strong>Q34.</strong> ATL schema: “if q holds, coalition C can keep p false forever.”</summary>

Method:<br>1. Put the condition first: q ⇒ ...<br>2. Use ⟪C⟫ for coalition ability.<br>3. Use □¬p for “always not p.”<br>4. Combine as q ⇒ ⟪C⟫□¬p.<br>Reference: ⟪C⟫□¬p means coalition C can ensure p never becomes true.

</details>

<details>
<summary><strong>Q35.</strong> ATL schema: “if q holds, coalition C cannot force p next.”</summary>

Method:<br>1. Express the ability claim as ⟪C⟫○p.<br>2. Negate that ability: ¬⟪C⟫○p.<br>3. Attach the condition: q ⇒ ¬⟪C⟫○p.<br>Reference: ¬⟪C⟫○p means C does not have a strategy guaranteeing p at the next step.

</details>

<details>
<summary><strong>Q36.</strong> Construct a concurrent-game-structure transition.</summary>

Method:<br>1. Start at state s.<br>2. List each agent’s available actions d(aᵢ,s).<br>3. Form a joint action tuple (α₁,...,αₙ).<br>4. Check each αᵢ is available.<br>5. Apply τ(s,α₁,...,αₙ) to get the next state.<br>Reference: a concurrent game structure uses available-action function d and transition function τ.

</details>

<details>
<summary><strong>Q37.</strong> Discriminator: individual action vs joint action in a concurrent game structure.</summary>

Use this discriminator:<br>1. αᵢ is one agent’s chosen action.<br>2. (α₁,...,αₙ) is the simultaneous joint action profile.<br>3. The transition function consumes the joint action, not just one agent’s action.<br>Reference: τ(s,α₁,...,αₙ) returns the next state.

</details>

<details>
<summary><strong>Q38.</strong> When is an ATL ability claim satisfied at a high level?</summary>

Method:<br>1. Identify the coalition C.<br>2. Ask whether C has a strategy for choosing its actions.<br>3. Check that following that strategy guarantees φ despite the concurrent setting.<br>Reference: cooperation modality = “this coalition has the power to ensure this outcome.”

</details>

<details>
<summary><strong>Q43.</strong> Use the bad-path method for model checking.</summary>

Method:<br>1. Represent the system as a transition model.<br>2. Represent the negation of the desired property as another transition model/automaton.<br>3. Combine the two.<br>4. Check whether the combined model has any accepting/violating path.<br>5. Empty combination means no bad path is possible.<br>Reference: model checking can be framed as searching for paths satisfying ¬φ.

</details>

<details>
<summary><strong>Q44.</strong> Negate a safety property into a bad-path property.</summary>

Method:<br>1. If the desired claim is “p never happens,” write □¬p.<br>2. Negate it to describe failure: ◇p.<br>3. Model-check by searching for a path that eventually reaches p.<br>Reference: bad paths are paths through the model that violate the property.

</details>

<details>
<summary><strong>Q45.</strong> Discriminator: desired property vs bad path.</summary>

Use this discriminator:<br>1. The desired property states what must hold in all allowed behaviour.<br>2. The bad-path property is its negation.<br>3. If the bad path exists, verification fails.<br>Reference: the negated property represents violating paths through the system.

</details>

<details>
<summary><strong>Q46.</strong> Why use Modular Interpreted Systems instead of one full global model?</summary>

<details>

Method:<br>1. Describe each agent locally.<br>2. Include what actions each agent can take.<br>3. Include what information each agent has.<br>4. Combine agents by unfolding their concurrent actions during model checking.<br>Reference: MIS represents the multi-agent system implicitly rather than as one manually built global transition system.

</details>

<details>
<summary><strong>Q50.</strong> Compute the available joint actions d(s) in an MIS.</summary>

</details>

<details>
<summary><strong>Q51.</strong> Build one global transition from local MIS transition functions.</summary>

Method:<br>1. Start with global state s = ⟨s₁,...,sₙ,s_env⟩.<br>2. Choose a joint action α ∈ d(s).<br>3. Apply each local transition function oᵢ(sᵢ,α).<br>4. Apply the environment transition if present.<br>5. Combine the results into the next global state.<br>Reference: next state = ⟨o₁(s₁,α),...,oₙ(sₙ,α),o_env(s_env,α)⟩.

</details>

<summary><strong>Q52.</strong> How does model checking unfold an MIS?</summary>
