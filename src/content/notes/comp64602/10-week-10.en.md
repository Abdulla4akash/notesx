# 1. Big picture: Gwendolen and BDI programming

## 1.1 What Gwendolen is

Gwendolen is a **Beliefs–Desires–Intentions** programming language used for agent programming. The lecturer compares it with **Jason**, which is more widely used, but explains that Gwendolen is used in this course because it was developed by the lecturer and has a specific application to **verification of agent programs**, which the course will look at later.

### Key concept: BDI programming

**Intuition:**  
A BDI agent is programmed in terms of what it believes about the world, what goals it has, and which current lines of action it is pursuing.

**In this lecture’s Gwendolen setting:**

- **Beliefs** are facts in the agent’s belief base, such as `empty`, `at(5, 5)`, or `rubble(5, 5)`.
- **Goals** are things the agent wants to bring about or perform, such as `pickup [achieve]` or `holding(rubble) [achieve]`.
- **Intentions** are the active stacks of work the agent is currently processing.
- **Plans** specify how the agent should react to events under certain conditions.

## 1.2 Why BDI/Gwendolen is “high-level”

Gwendolen is intended to be **high-level**: it is concerned with the agent’s big decision-making, not low-level control. For example, a plan may contain an action such as:

```gwendolen
pick-up(block)
```

or:

```gwendolen
lift_rubble
```

The Gwendolen program does not specify the robotics details of moving motors, controlling a gripper, or detecting forces. Those details are assumed to be handled by lower-level control code in the environment.

### Key concept: action

**Definition:**  
An **action** is a deed in a plan with no `+`, `-`, or `!` decoration. It is something the agent asks the environment or low-level system to do.

Examples:

```gwendolen
pick-up(block)
move_to(X, Y)
lift_rubble
print(done)
```

Actions are contrasted with deeds that add or remove beliefs/goals, such as:

```gwendolen
+pickup
-empty
+!holding(rubble) [achieve]
```

## 1.3 Why BDI/Gwendolen is “declarative”

The language is meant to focus on expressing the problem rather than spelling out every detail of the solution. The programmer writes plans in terms of beliefs, goals, and conditions.

Example from the intro slides:

```gwendolen
+!empty : {B holding(block)} <-
    send(co-worker, take(co-worker, block));

+!empty : {B holding(A)} <-
    send(co-worker, take(co-worker, A));

+!holding(block) : {B empty, B on_ground(block)} <-
    pick-up(block);

+on_ground(block) : {~G holding(block)} <-
    +!holding(block);
```

Interpretation:

- If the agent gets a goal to have empty hands and believes it is holding a block, it sends a message to a co-worker asking them to take it.
- If the agent gets a goal to have empty hands and believes it is holding something `A`, it sends a message to a co-worker asking them to take `A`.
- If the agent gets a goal to hold a block and believes its hands are empty and the block is on the ground, it performs a pick-up action.
- If the agent perceives that a block is on the ground and does not already have the goal of holding it, it adopts the goal of holding it.

## 1.4 “Declarative … or not”

The lecturer warns that real programs can become less clean. The satellite-control example on the slides has several plans for the same event:

```gwendolen
+!maintaining(Pos) [achieve] : {B in_position(Pos), B assuming_formation(F),
    ~B aborted(Reason), ~B broken(X)} <-
        perf(maintain_path),
        +maintaining(Pos),
        +!cleanup [perform];

+!maintaining(Pos) [achieve] : {~B in_position(Pos), B assuming_formation(F),
    ~B aborted(Reason), ~B broken(X)} <-
        +!in_position(Pos) [achieve],
        perf(maintain_path),
        +maintaining(Pos),
        +!cleanup [perform];

+!maintaining(Pos) [achieve] : {B broken(X), ~B aborted(Reason)} <-
        *fixed(X),
        -fixed(X);

+!maintaining(Pos) [achieve] : {~B assuming_formation(F)} <-
        -!maintaining(Pos) [achieve];

+!maintaining(Pos) [achieve] : {B aborted(Reason), B assuming_formation(F)} <-
        -!assuming_formation(F) [perform];
```

The point is not the exact satellite-control details. The point is that real agent programs may require many different plans for the same goal depending on beliefs such as:

- whether the agent is already in position;
- whether it is assuming a formation;
- whether the mission has been aborted;
- whether a component is broken.

> **Exam flag — important caveat:** BDI languages aim to be declarative, but non-trivial applications can still require many plans and complicated guards.

# 2. Basic Gwendolen program structure

A Gwendolen program has sections such as:

```gwendolen
GWENDOLEN

:name: ag1

:Initial Beliefs:
empty

:Initial Goals:
pickup [achieve]

:Plans:
+!pickup [achieve] : {B empty} <-
    +pickup,
    -empty,
    print(done);
```

This is the simple pick-up example from the slides.

## 2.1 Components of a program

### `:name:`

Names the agent.

Example:

```gwendolen
:name: ag1
```

The lecturer notes that a program can contain more than one agent, so agents need names.

### `:Initial Beliefs:`

Facts initially in the agent’s belief base.

Example:

```gwendolen
empty
```

The agent initially believes it is empty-handed.

### `:Initial Goals:`

Goals initially given to the agent.

Example:

```gwendolen
pickup [achieve]
```

The agent starts with the goal of achieving `pickup`.

### `:Plans:`

Rules saying how to react to events.

General form:

```gwendolen
trigger : {guard} <- deed1, deed2, ..., deedN;
```

For example:

```gwendolen
+!pickup [achieve] : {B empty} <-
    +pickup,
    -empty,
    print(done);
```

Interpretation:

- Trigger: the goal `pickup [achieve]` has appeared.
- Guard: the agent believes `empty`.
- Deeds:
  - add belief `pickup`;
  - remove belief `empty`;
  - print `done`.

## 2.2 Plan syntax

### Trigger event

The part before the colon:

```gwendolen
+!pickup [achieve]
```

Meaning:

- `+` means something has appeared or been added.
- `!` means it is a goal.
- No `!` means it is a belief.
- `-` would mean something has disappeared or been removed.

Examples:

```gwendolen
+!holding(rubble) [achieve]
```

A goal to achieve `holding(rubble)` has appeared.

```gwendolen
+at(X, Y)
```

The belief `at(X, Y)` has appeared.

```gwendolen
-rubble(X, Y)
```

The belief `rubble(X, Y)` has disappeared.

### Guard

The condition inside curly braces:

```gwendolen
{B empty}
```

Meaning: this plan applies only if the agent believes `empty`.

Common guard notation from the lectures:

```gwendolen
B p
```

The agent believes `p`.

```gwendolen
~B p
```

The agent does not believe `p`.

```gwendolen
G g [achieve]
```

The agent has goal `g [achieve]`.

```gwendolen
True
```

The guard is always true.

### Deeds

The right-hand side after `<-`.

Examples:

```gwendolen
+pickup
-empty
print(done)
move_to(X, Y)
+!holding(rubble) [achieve]
```

Deeds can:

- add beliefs;
- remove beliefs;
- add goals;
- remove goals;
- call actions;
- wait for a belief;
- lock/unlock an intention.

# 3. Achieve goals vs perform goals

The lectures distinguish **achieve goals** and **perform goals**. This is a major concept and appears in both the introductory lecture and the events/intentions lecture.

## 3.1 Perform goals

### Definition

When an agent adds a **perform goal**, it:

1. searches for a plan for that goal;
2. executes the plan;
3. drops the goal.

Perform goals are used mainly for structuring code, like subroutines or methods.

Example:

```gwendolen
+!goto(Room) [perform] : {T} <-
    planRoute(Room, Route),
    follow(Route),
    enter(Room);
```

The goal is not maintained until some belief becomes true. The agent just performs the plan and then drops it.

### Intuition

A perform goal means: “do this procedure.”

It does not mean: “keep trying until the world satisfies this condition.”

## 3.2 Achieve goals

### Definition

When an agent adds an **achieve goal**, it:

1. checks whether the goal is already achieved by looking for a corresponding belief in the belief base;
2. if not achieved, searches for and executes a plan for that goal;
3. after executing the plan, checks again whether it now believes the goal has been achieved;
4. if it still lacks the corresponding belief, it searches for a plan again.

Example:

```gwendolen
holding(rubble) [achieve]
```

This goal remains active until the agent believes:

```gwendolen
holding(rubble)
```

### Intuition

An achieve goal means: “make this true, and keep trying until I believe it is true.”

## 3.3 Worked example: simple `pickup [achieve]`

Program:

```gwendolen
GWENDOLEN

:name: ag1

:Initial Beliefs:
empty

:Initial Goals:
pickup [achieve]

:Plans:
+!pickup [achieve] : {B empty} <-
    +pickup,
    -empty,
    print(done);
```

Execution:

1. The agent starts with initial belief:

   ```gwendolen
   empty
   ```

2. The initial goal is:

   ```gwendolen
   pickup [achieve]
   ```

3. Because this is an initial goal, Gwendolen creates a start event that adds the goal.

4. The plan for `+!pickup [achieve]` applies because the guard is true:

   ```gwendolen
   {B empty}
   ```

5. The plan executes:

   ```gwendolen
   +pickup
   -empty
   print(done)
   ```

6. The belief base changes from:

   ```gwendolen
   empty
   ```

   to:

   ```gwendolen
   pickup
   ```

7. The program prints:

   ```text
   done
   ```

The lecturer notes that this is not realistic robotics: a robot does not pick something up just by adding a belief. It is a small example used to show the mechanics.

# 4. The Gwendolen reasoning cycle

The reasoning-cycle diagram appears in both the intro slides and the events/intentions slides. The transcript stresses the order of the stages because it explains the later “extra move” in the rubble example.

## 4.1 Algorithm: Gwendolen reasoning cycle

The cycle is:

1. **Stage A — Select a current intention or sleep the agent.**
   - Gwendolen chooses one intention to work on.
   - By default, it cycles through intentions in turn.
   - If there are no usable intentions, the agent may sleep or move to perception.

2. **Decision — Are there any unsuspended non-empty intentions?**
   - If yes, continue to planning.
   - If no, go to perception.

3. **Stage B — Find all plans applicable to the current intention.**
   - This happens when the current intention needs planning, usually because its top deed is `no plan yet`.

4. **Stage C — Pick a plan and apply it.**
   - If multiple plans apply, one is selected.
   - Applying a plan usually adds its deeds to the top of the intention stack.

5. **Decision — Is the current intention empty?**
   - If yes, go to perception.
   - If no, execute the top deed.

6. **Stage D — Execute the top deed on the current intention.**
   - This may be an action, belief addition/removal, goal addition/removal, wait, lock, etc.

7. **Stage E — Get new perception and messages.**
   - Perceptions may create new intentions.
   - For example, perceiving `at(1, 1)` creates an intention to add the belief `at(1, 1)`.

8. **Stage F — Process new messages.**
   - Messages can also create new events/intentions.

9. Return to Stage A.

## 4.2 Why the ordering matters

The key order is:

```text
select intention
→ plan if needed
→ execute top deed
→ get perception/messages
→ select another intention
```

This ordering means:

- Actions can cause new perceptions.
- Perceptions do not automatically become beliefs immediately.
- Perceptions create intentions to add beliefs.
- The belief is added only when that intention is processed.

This causes a lag between an action producing a percept and the corresponding belief appearing in the belief base.

> **Exam flag — high value:** The perception lag is central to the extra `move_to(5, 5)` in the rubble example.

# 5. Events

## 5.1 Definition of an event

Plans are triggered by **events**. An event is something changing: a belief or goal being added or removed.

Formal examples:

```gwendolen
+belief
-belief
+!goal [achieve]
-!goal [achieve]
```

A plan does not fire just because a condition is true. It fires when the triggering event occurs and the guard is satisfied.

## 5.2 Special `start` event

There is a special `start` event used for:

1. an agent’s initial goals;
2. information coming from perception that is to be turned into beliefs.

Example initial goal intention:

```text
Trigger: +start
Deed:    +!holding(rubble)
```

Example perceptual intention:

```text
Trigger: +start
Deed:    +at(1, 1)
```

The `start` event is used because the initial goal or perceptual belief needs to be inserted into the agent’s belief/goal machinery.

## 5.3 Common mistake: confusing events with conditions

The lecture gives an obstacle example:

```gwendolen
+obstacle : {T} <- turn_90, forward.
```

Scenario:

1. The robot moves toward a corner.
2. It detects an obstacle.
3. The belief `obstacle` appears.
4. The event `+obstacle` triggers the plan.
5. The robot turns 90 degrees.
6. The robot moves forward.
7. It hits the other wall.

Why does it not turn again?

Because while the robot was turning, the obstacle belief never disappeared and reappeared. The belief stayed true. Since no new `+obstacle` event occurred, the plan did not trigger again.

> **Exam flag — common mistake:** Plans are triggered by changes/events, not by facts merely being true. Use guards for facts that are true; use triggers for things that have just changed.

# 6. Intentions

## 6.1 Definition of an intention

An **intention** is a stack of tuples. Each tuple contains:

```text
(trigger event, deed, unifier)
```

From the slides:

```text
Trigger | Deed | Unifier
```

The lecturer sometimes simplifies later examples by omitting the unifier column, but it is still part of the underlying structure.

### Key concept: unifier

**Intuition:**  
The unifier records variable bindings produced when matching a plan to an event and guard.

Example:

```text
{Room = room1}
```

means the variable `Room` has been unified with the concrete value `room1`.

### Key concept: deed

A **deed** is the current instruction to execute in a tuple. It may be an action, a belief update, a goal update, `no plan yet`, `null`, wait, lock, etc.

## 6.2 Special deed: `no plan yet`

The slides abbreviate this as:

```text
npy
```

The transcript sometimes renders it as “NPI” or “no plan yet.”

Definition:

```text
no plan yet
```

means the agent has an event on the intention stack but has not yet decided how to handle it. When the top deed is `no plan yet`, Gwendolen searches for applicable plans.

## 6.3 Special deed: `null`

For perform goals, the deed that originally added the perform goal is replaced by `null`.

Meaning:

- The perform goal becomes a new event to be planned.
- The old deed is not kept around for achievement checking.
- Once the perform goal’s plan has been executed, it disappears.

This differs from achieve goals, where the goal deed remains so Gwendolen can later check whether the goal has actually been achieved.

# 7. Worked example: `clean` intention stack

The events lecture uses this program:

```gwendolen
GWENDOLEN

:name: temp

:Initial Beliefs:

:Initial Goals:
clean [achieve]

:Plans:
+!clean [achieve] : {B dirty(Room)} <-
    +!goto(Room) [perform],
    +!vacuum(Room) [perform];

+!goto(Room) [perform] : {T} <-
    planRoute(Room, Route),
    follow(Route),
    enter(Room);
```

Assume the agent believes:

```gwendolen
dirty(room1)
```

## 7.1 Initial intention

Initial goals use the `start` event.

```text
Trigger     Deed                  Unifier
+start      +!clean [achieve]      {}
```

The top deed is to add the achieve goal `clean`.

Because `clean` is an achieve goal, the agent first checks whether it already believes:

```gwendolen
clean
```

It does not, so the goal is added.

## 7.2 Adding the achieve goal creates a new event

The stack becomes:

```text
Trigger              Deed             Unifier
+!clean [achieve]    no plan yet       {}
+start               +!clean [achieve] {}
```

The top row means:

- the event is acquiring the goal `clean [achieve]`;
- the agent has not yet chosen a plan for that event.

## 7.3 Applying the `clean` plan

The plan:

```gwendolen
+!clean [achieve] : {B dirty(Room)} <-
    +!goto(Room) [perform],
    +!vacuum(Room) [perform];
```

applies because the agent believes:

```gwendolen
dirty(room1)
```

So unification gives:

```text
Room = room1
```

`no plan yet` is removed and the plan deeds are placed on top of the intention:

```text
Trigger              Deed                         Unifier
+!clean [achieve]    +!goto(Room) [perform]        {Room = room1}
+!clean [achieve]    +!vacuum(Room) [perform]      {Room = room1}
+start               +!clean [achieve]             {}
```

## 7.4 Handling a perform goal

When the perform goal `+!goto(Room) [perform]` becomes an event, the original deed is replaced by `null`.

```text
Trigger                  Deed           Unifier
+!goto(Room) [perform]   no plan yet     {Room = room1}
+!clean [achieve]        null            {Room = room1}
+!clean [achieve]        +!vacuum(Room)  {Room = room1}
+start                   +!clean         {}
```

Interpretation:

- The agent must now plan how to perform `goto(room1)`.
- The original perform-goal deed has become `null`.
- Once the `goto` plan’s deeds are executed, the system continues down the stack.
- The perform goal does not remain for achievement re-checking.

## 7.5 Applying the `goto` plan

The plan:

```gwendolen
+!goto(Room) [perform] : {T} <-
    planRoute(Room, Route),
    follow(Route),
    enter(Room);
```

is always applicable because its guard is `{T}`.

The deeds are:

```gwendolen
planRoute(Room, Route)
follow(Route)
enter(Room)
```

These are actions. As they execute, they are removed from the intention unless they generate further events.

# 8. Multiple intentions

## 8.1 What starts an intention?

Each event being reacted to starts an intention. This includes:

- acquiring an initial goal;
- perceiving information that becomes a belief;
- adding a belief;
- removing a belief;
- adding or removing goals.

The slides state that each event being reacted to, such as acquiring an initial goal or belief, starts an intention. Gwendolen cycles through intentions and handles the top deed on each in turn.

## 8.2 Why multiple intentions are useful

They allow the agent to react to new events while pursuing existing goals.

Example intuition:

- The robot may be pursuing a rubble-clearing goal.
- A new obstacle may be perceived.
- The obstacle event can start its own intention.
- The agent can react to the obstacle without abandoning the original goal.

## 8.3 Why multiple intentions cause problems

Because intentions are interleaved, the agent may switch away from a partially completed sequence.

This can be good for responsiveness, but bad when:

- a sequence of belief updates must happen together;
- the current intention must wait for perceptual processing;
- a mode switch is half-complete;
- one intention sees the belief base in an intermediate state.

The rest of the lecture cluster builds up to two mechanisms for controlling this:

```text
waiting
locking
```

# 9. Baseline worked example: pick up rubble

The same rubble program is used repeatedly across the lectures.

## 9.1 Program

```gwendolen
GWENDOLEN

:name: robot

:Initial Beliefs:
possible_rubble(1, 1)
possible_rubble(3, 3)
possible_rubble(5, 5)

:Initial Goals:
holding(rubble) [achieve]

:Plans:
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <-
        move_to(X, Y);

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        lift_rubble;

+holding(rubble) :
    {True} <-
        print(done);
```

## 9.2 Intended environment

The environment used in the lecture behaves as follows:

```text
No rubble at (1, 1)
No rubble at (3, 3)
Rubble at (5, 5)
```

So the natural expected behaviour is:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
print(done)
```

But the actual behaviour includes an extra move:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
move_to(5, 5)
print(done)
```

The extra `move_to(5, 5)` is caused by the interaction between achieve goals, perception lag, and multiple intentions.

## 9.3 Why the first plan repeatedly applies

The initial goal is:

```gwendolen
holding(rubble) [achieve]
```

The goal remains active until the agent believes:

```gwendolen
holding(rubble)
```

The plan for achieving the goal is:

```gwendolen
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <-
        move_to(X, Y);
```

So whenever the achievement goal is reconsidered, the agent looks for a square such that:

```gwendolen
possible_rubble(X, Y)
```

is believed, and:

```gwendolen
no_rubble(X, Y)
```

is not believed.

If `(1, 1)` has been ruled out by adding `no_rubble(1, 1)`, the next applicable square is `(3, 3)`. If `(3, 3)` has been ruled out, the next applicable square is `(5, 5)`.

## 9.4 Detailed trace: start and first move

Initial intention:

```text
Trigger     Deed
+start      +!holding(rubble)
```

The initial goal is added. Because it is an achieve goal and the agent does not yet believe `holding(rubble)`, Gwendolen creates an event for the new goal:

```text
Trigger                  Deed
+!holding(rubble)        no plan yet
+start                   +!holding(rubble)
```

The first plan applies with:

```text
X = 1
Y = 1
```

because:

```gwendolen
possible_rubble(1, 1)
```

is believed and:

```gwendolen
no_rubble(1, 1)
```

is not believed.

The intention becomes:

```text
Trigger                  Deed
+!holding(rubble)        move_to(1, 1)
+start                   +!holding(rubble)
```

The action executes:

```gwendolen
move_to(1, 1)
```

Then perception occurs.

## 9.5 Perception after moving to `(1, 1)`

After moving, the agent perceives it is at `(1, 1)`. Perception creates an intention:

```text
Trigger     Deed
+start      +at(1, 1)
```

Processing that intention adds the belief:

```gwendolen
at(1, 1)
```

Adding this belief creates a new event:

```text
Trigger     Deed
+at(1, 1)   no plan yet
```

Now the plan:

```gwendolen
+at(X, Y) : {~B rubble(X, Y)} <- +no_rubble(X, Y);
```

applies, because at `(1, 1)` the agent does not believe there is rubble.

So it adds:

```gwendolen
no_rubble(1, 1)
```

The belief base now includes:

```gwendolen
at(1, 1)
no_rubble(1, 1)
```

## 9.6 Moving to `(3, 3)`

The achieve goal `holding(rubble)` is still not satisfied, because the agent does not believe:

```gwendolen
holding(rubble)
```

When that intention is reconsidered, the first plan is applied again. `(1, 1)` is no longer suitable because:

```gwendolen
no_rubble(1, 1)
```

is now believed.

So the agent moves to `(3, 3)`:

```gwendolen
move_to(3, 3)
```

Perception then creates intentions to:

```gwendolen
+at(3, 3)
-at(1, 1)
```

The agent eventually adds:

```gwendolen
at(3, 3)
no_rubble(3, 3)
```

and removes:

```gwendolen
at(1, 1)
```

The belief base includes:

```gwendolen
no_rubble(1, 1)
at(3, 3)
no_rubble(3, 3)
```

## 9.7 Moving to `(5, 5)`

The achieve goal is still unsatisfied. `(1, 1)` and `(3, 3)` are ruled out. The next possible square is `(5, 5)`.

The agent executes:

```gwendolen
move_to(5, 5)
```

Perception creates intentions to:

```gwendolen
+at(5, 5)
-at(3, 3)
+rubble(5, 5)
```

The belief base eventually includes:

```gwendolen
no_rubble(1, 1)
no_rubble(3, 3)
at(5, 5)
rubble(5, 5)
```

## 9.8 Lifting rubble

The event:

```gwendolen
+rubble(5, 5)
```

triggers this plan:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- lift_rubble;
```

The guard holds because the agent believes:

```gwendolen
at(5, 5)
```

So the action executes:

```gwendolen
lift_rubble
```

After this, perception eventually informs the agent that it is holding rubble:

```gwendolen
holding(rubble)
```

But this is not added to the belief base immediately. It first becomes an intention:

```text
Trigger     Deed
+start      +holding(rubble)
```

## 9.9 Why the extra `move_to(5, 5)` happens

Before the intention to add:

```gwendolen
holding(rubble)
```

is processed, Gwendolen may return to the original achieve-goal intention:

```gwendolen
+!holding(rubble) [achieve]
```

At that moment:

- the agent has done `lift_rubble`;
- the environment has generated a perception that it is holding rubble;
- but the belief `holding(rubble)` is not yet in the belief base.

So the achieve goal is still considered unsatisfied.

The first plan still applies for `(5, 5)` because:

```gwendolen
possible_rubble(5, 5)
```

is believed, and:

```gwendolen
no_rubble(5, 5)
```

is not believed.

So the agent executes:

```gwendolen
move_to(5, 5)
```

again.

Then the perception intention is processed, adding:

```gwendolen
holding(rubble)
```

Now the `+holding(rubble)` event triggers:

```gwendolen
+holding(rubble) : {True} <- print(done);
```

and the agent prints:

```text
done
```

> **Exam flag — important mechanism:** The extra move is not random. It follows from the reasoning-cycle order and the fact that perceptions become beliefs only after their own intentions are processed.

# 10. Plan guards and reasoning rules

This lecture extends the rubble example by moving some guard logic into reasoning rules, then showing lists, recursion, and goals in guards.

## 10.1 Plan guards

A **plan guard** is the condition between `{ ... }` that must hold for the plan to apply.

Example:

```gwendolen
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <-
        move_to(X, Y);
```

The guard says:

- the agent believes there might be rubble at `(X, Y)`;
- the agent does not believe there is no rubble at `(X, Y)`.

### Key concept: guard vs trigger

**Trigger:** what just happened.

```gwendolen
+!holding(rubble) [achieve]
```

**Guard:** what must currently be true.

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

The trigger is event-based; the guard is state-based.

## 10.2 Reasoning rules

Reasoning rules are used inside guards to derive useful predicates from beliefs.

### Formal syntax

```gwendolen
Head :- Body;
```

The transcript says this is Prolog-like and similar to Datalog. The `:-` is an implication-like arrow: if the body holds, the head holds.

## 10.3 Reasoning-rule version of the rubble program

Program from the slides:

```gwendolen
GWENDOLEN

:name: robot

:Initial Beliefs:
possible_rubble(1, 1)
possible_rubble(3, 3)
possible_rubble(5, 5)

:Reasoning Rules:
square_to_check(X, Y) :-
    possible_rubble(X, Y),
    ~no_rubble(X, Y);

:Initial Goals:
holding(rubble) [achieve]

:Plans:
+!holding(rubble) [achieve] :
    {B square_to_check(X, Y)} <-
        move_to(X, Y);

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        lift_rubble;

+holding(rubble) :
    {True} <-
        print(done);
```

## 10.4 Meaning of `square_to_check`

Reasoning rule:

```gwendolen
square_to_check(X, Y) :-
    possible_rubble(X, Y),
    ~no_rubble(X, Y);
```

### Intuition

A square is worth checking if:

1. it is a square where rubble might be; and
2. we have not already established that there is no rubble there.

### Formal reading

For any `X, Y`:

```text
square_to_check(X, Y)
```

holds if:

```text
possible_rubble(X, Y)
```

holds and:

```text
no_rubble(X, Y)
```

does not hold.

This allows the plan guard to be cleaner:

```gwendolen
{B square_to_check(X, Y)}
```

instead of:

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

The behaviour is the same as the previous version.

## 10.5 Using lists in guards

The next version stores all possible rubble locations in one list instead of three separate beliefs.

Initial belief:

```gwendolen
possible_rubble([sq(1, 1), sq(3, 3), sq(5, 5)])
```

Reasoning rules:

```gwendolen
square_to_check(X, Y) :-
    possible_rubble(L),
    check_rubble(L, X, Y);

check_rubble([sq(X, Y) | T], X, Y) :-
    ~no_rubble(X, Y);

check_rubble([sq(X, Y) | T], X1, Y1) :-
    no_rubble(X, Y),
    check_rubble(T, X1, Y1);
```

The rest of the plans stay the same:

```gwendolen
+!holding(rubble) [achieve] :
    {B square_to_check(X, Y)} <-
        move_to(X, Y);

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        lift_rubble;

+holding(rubble) :
    {True} <-
        print(done);
```

## 10.6 List notation

The lecture uses Prolog-style list notation:

```gwendolen
[Head | Tail]
```

In this case:

```gwendolen
[sq(X, Y) | T]
```

means:

- the first element of the list is `sq(X, Y)`;
- the rest of the list is `T`.

Example:

```gwendolen
[sq(1, 1), sq(3, 3), sq(5, 5)]
```

matches:

```gwendolen
[sq(X, Y) | T]
```

with:

```text
X = 1
Y = 1
T = [sq(3, 3), sq(5, 5)]
```

## 10.7 Recursive derivation: first square

Initial list:

```gwendolen
possible_rubble([sq(1, 1), sq(3, 3), sq(5, 5)])
```

No `no_rubble` beliefs yet.

To derive:

```gwendolen
square_to_check(X, Y)
```

use:

```gwendolen
square_to_check(X, Y) :- possible_rubble(L), check_rubble(L, X, Y);
```

So `L` becomes:

```gwendolen
[sq(1, 1), sq(3, 3), sq(5, 5)]
```

Now evaluate:

```gwendolen
check_rubble([sq(1, 1), sq(3, 3), sq(5, 5)], X, Y)
```

The base-case rule applies:

```gwendolen
check_rubble([sq(X, Y) | T], X, Y) :-
    ~no_rubble(X, Y);
```

because the agent does not believe:

```gwendolen
no_rubble(1, 1)
```

Therefore:

```text
X = 1
Y = 1
```

and:

```gwendolen
square_to_check(1, 1)
```

is derived.

## 10.8 Recursive derivation: after `(1, 1)` has been checked

Suppose the belief base contains:

```gwendolen
no_rubble(1, 1)
```

Now the base case for the head of the list fails, because:

```gwendolen
~no_rubble(1, 1)
```

is false.

The recursive rule applies:

```gwendolen
check_rubble([sq(X, Y) | T], X1, Y1) :-
    no_rubble(X, Y),
    check_rubble(T, X1, Y1);
```

With the list:

```gwendolen
[sq(1, 1), sq(3, 3), sq(5, 5)]
```

this gives:

```text
X = 1
Y = 1
T = [sq(3, 3), sq(5, 5)]
```

Since:

```gwendolen
no_rubble(1, 1)
```

is believed, recurse on:

```gwendolen
check_rubble([sq(3, 3), sq(5, 5)], X1, Y1)
```

If the agent does not believe:

```gwendolen
no_rubble(3, 3)
```

then the base case now returns:

```text
X1 = 3
Y1 = 3
```

So the next square to check is:

```gwendolen
square_to_check(3, 3)
```

## 10.9 Recursive derivation: after `(1, 1)` and `(3, 3)` have been checked

Beliefs:

```gwendolen
no_rubble(1, 1)
no_rubble(3, 3)
```

The first list element is skipped by recursion. The second list element is also skipped by recursion. The remaining list is:

```gwendolen
[sq(5, 5)]
```

If the agent does not believe:

```gwendolen
no_rubble(5, 5)
```

then the base case returns:

```text
X = 5
Y = 5
```

So:

```gwendolen
square_to_check(5, 5)
```

is derived.

> **Exam flag — useful technique:** Reasoning rules can use lists and recursion to process structured data in plan guards. The lecturer explicitly connects this to Datalog/Prolog-style unification.

## 10.10 Goals in guards

The lecture also shows that goals can appear in guards.

Program:

```gwendolen
GWENDOLEN

:name: robot

:Initial Beliefs:
possible_rubble(1, 1)
possible_rubble(3, 3)
possible_rubble(5, 5)

:Initial Goals:
rubble(2, 2) [achieve]

:Plans:
+!rubble(2, 2) [achieve] : {True} <-
    +!holding(rubble) [achieve],
    move_to(2, 2),
    drop;

+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <-
        move_to(X, Y);

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y), G holding(rubble) [achieve]} <-
        lift_rubble;
```

## 10.11 Meaning of the goal guard

The key plan is:

```gwendolen
+rubble(X, Y) :
    {B at(X, Y), G holding(rubble) [achieve]} <-
        lift_rubble;
```

This says:

- if rubble appears at `(X, Y)`;
- and the agent believes it is at `(X, Y)`;
- and the agent currently has the goal `holding(rubble) [achieve]`;
- then it should lift the rubble.

### Why add `G holding(rubble) [achieve]`?

It prevents the robot from lifting rubble just because it sees rubble. It lifts rubble only when lifting rubble is relevant to its current goal.

## 10.12 Worked example: moving rubble to `(2, 2)`

Initial goal:

```gwendolen
rubble(2, 2) [achieve]
```

Plan:

```gwendolen
+!rubble(2, 2) [achieve] : {True} <-
    +!holding(rubble) [achieve],
    move_to(2, 2),
    drop;
```

Execution outline:

1. The agent adopts the subgoal:

   ```gwendolen
   holding(rubble) [achieve]
   ```

2. It searches possible rubble locations:

   ```gwendolen
   possible_rubble(1, 1)
   possible_rubble(3, 3)
   possible_rubble(5, 5)
   ```

3. It checks `(1, 1)` and `(3, 3)`, finding no rubble.

4. It reaches `(5, 5)` and perceives:

   ```gwendolen
   rubble(5, 5)
   ```

5. Because the agent is at `(5, 5)` and has the goal `holding(rubble) [achieve]`, the guarded rubble plan applies:

   ```gwendolen
   lift_rubble
   ```

6. After the rubble is lifted, the higher-level goal plan continues:

   ```gwendolen
   move_to(2, 2)
   drop
   ```

7. The intended result is that rubble is placed at `(2, 2)`.

The transcript notes that the same perception lag can still cause an extra move to `(5, 5)` before moving to `(2, 2)`.

# 11. Controlling multiple intentions: waiting

The next lecture introduces two mechanisms for controlling multiple intentions: **waiting** and **locking**. Waiting is used first to stop the rubble-searching agent from “charging ahead” to another movement before it has processed the current square.

## 11.1 Problem being solved

In the baseline rubble program, the achieve goal `holding(rubble)` keeps being reconsidered. Because perceptions and belief updates are processed through separate intentions, the agent can choose the original goal intention again before the perceptual consequences of the previous action have been fully processed.

Waiting gives the programmer a way to say:

```text
Do not continue this intention until a particular belief is true.
```

## 11.2 Wait syntax

The wait deed is written with `*`.

Example:

```gwendolen
*checked(X, Y)
```

Meaning:

```text
Wait until the agent believes checked(X, Y).
```

The transcript says “star means wait for.”

### Important detail

Waiting suspends or skips that intention until the waited-for belief appears. It does not stop the whole agent. Other intentions can still run.

## 11.3 Waiting version of the rubble program

The slides show the waiting version as:

```gwendolen
:Plans:

+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <-
        move_to(X, Y),
        *checked(X, Y);

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y),
        +checked(X,Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        lift_rubble,
        +checked(X, Y);

+holding(rubble) :
    {True} <-
        print(done);
```

[UNCLEAR] The first waiting slide includes `B search_mode` in the guard, while the subsequent worked waiting trace on the slides focuses on the wait/check mechanism and often shows the simpler guard without `B search_mode`. The wait mechanism itself is the important point.

## 11.4 Meaning of `checked(X, Y)`

The new belief:

```gwendolen
checked(X, Y)
```

means that the agent has finished processing square `(X, Y)`.

A square becomes checked in either of two cases:

1. The agent arrives there and does not believe there is rubble:

   ```gwendolen
   +at(X, Y) : {~B rubble(X, Y)} <-
       +no_rubble(X, Y),
       +checked(X,Y);
   ```

2. The agent sees rubble there and is at the square:

   ```gwendolen
   +rubble(X, Y) : {B at(X, Y)} <-
       lift_rubble,
       +checked(X, Y);
   ```

So after moving to a square, the main achievement intention waits until the appropriate perception-handling intention has marked the square as checked.

## 11.5 Worked trace: waiting at `(1, 1)`

Initial intention:

```text
Trigger     Deed
+start      +!holding(rubble)
```

After adding the achieve goal and planning:

```text
Trigger                  Deed
+!holding(rubble)        move_to(1, 1)
+!holding(rubble)        *checked(1, 1)
+start                   +!holding(rubble)
```

The agent executes:

```gwendolen
move_to(1, 1)
```

Now the top deed for the original intention is:

```gwendolen
*checked(1, 1)
```

The agent does not yet believe:

```gwendolen
checked(1, 1)
```

So that intention is skipped during intention selection.

Perception creates a new intention:

```text
Trigger     Deed
+start      +at(1, 1)
```

This adds:

```gwendolen
at(1, 1)
```

which creates:

```text
Trigger     Deed
+at(1, 1)   no plan yet
```

The plan for `+at(1, 1)` applies because there is no rubble there:

```gwendolen
+at(X, Y) : {~B rubble(X, Y)} <-
    +no_rubble(X, Y),
    +checked(X,Y);
```

So the deeds are:

```gwendolen
+no_rubble(1, 1)
+checked(1, 1)
```

After `+checked(1, 1)` is processed, the belief base includes:

```gwendolen
at(1, 1)
no_rubble(1, 1)
checked(1, 1)
```

Now the waiting deed succeeds:

```gwendolen
*checked(1, 1)
```

is removed, and the original achievement intention can continue.

## 11.6 What happens next

After `(1, 1)` is checked, the achieve goal is still not satisfied because the agent does not believe:

```gwendolen
holding(rubble)
```

So the agent repeats the achievement process, now avoiding `(1, 1)` because:

```gwendolen
no_rubble(1, 1)
```

is believed.

It moves to `(3, 3)`, waits for:

```gwendolen
checked(3, 3)
```

then eventually moves to `(5, 5)` and waits for:

```gwendolen
checked(5, 5)
```

## 11.7 Worked trace: waiting at `(5, 5)`

At the point before moving to `(5, 5)`, the belief base includes approximately:

```gwendolen
at(3, 3)
no_rubble(1, 1)
no_rubble(3, 3)
checked(1, 1)
checked(3, 3)
```

The intention contains:

```text
Trigger                  Deed
+!holding(rubble)        move_to(5, 5)
+!holding(rubble)        *checked(5, 5)
+start                   +!holding(rubble)
```

After moving, perception creates intentions to add/remove:

```gwendolen
+at(5, 5)
-at(3, 3)
+rubble(5, 5)
```

When `+rubble(5, 5)` is processed, this plan applies:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <-
    lift_rubble,
    +checked(X, Y);
```

So it executes:

```gwendolen
lift_rubble
+checked(5, 5)
```

The lift action also eventually produces a perception that the agent is holding rubble:

```gwendolen
holding(rubble)
```

After `checked(5, 5)` is believed, the waiting deed:

```gwendolen
*checked(5, 5)
```

succeeds and is removed.

Then, once `holding(rubble)` is believed, the plan:

```gwendolen
+holding(rubble) : {True} <- print(done);
```

prints:

```text
done
```

## 11.8 What waiting fixes

Waiting prevents the original achievement intention from racing ahead before the current square has been fully processed.

The transcript states that by putting the wait in, the program stops “charging ahead” and doing more movements until the processing needed for each square has completed.

### Key concept: waiting

**Intuition:**  
Waiting pauses one intention until a belief becomes true, while allowing other intentions to continue.

**Formal mechanism from the lecture:**  
A deed of the form:

```gwendolen
*p
```

means the intention should not proceed until the agent believes `p`.

# 12. Controlling multiple intentions: locking

Locking is the second mechanism for controlling multiple intentions. It is used when the programmer wants a sequence of deeds in one intention to happen without interleaving.

## 12.1 Mode-based rubble program

The locking section starts with an alternative way to control behaviour using modes:

```gwendolen
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <-
        move_to(X, Y);

+!holding(rubble) [achieve] :
    {B rubble(X, Y), B at(X, Y), B lift_mode} <-
        lift_rubble;

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        -search_mode,
        +lift_mode;

+holding(rubble) :
    {True} <-
        print(done);
```

## 12.2 Intended meaning of modes

There are two mode beliefs:

```gwendolen
search_mode
lift_mode
```

### `search_mode`

When the agent believes:

```gwendolen
search_mode
```

the first `holding(rubble)` plan applies:

```gwendolen
move_to(X, Y)
```

So the robot searches possible rubble locations.

### `lift_mode`

When the agent believes:

```gwendolen
lift_mode
```

and also believes:

```gwendolen
rubble(X, Y)
at(X, Y)
```

the second `holding(rubble)` plan applies:

```gwendolen
lift_rubble
```

So the robot lifts rubble.

## 12.3 The bug without locking

The mode-switch plan is:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <-
    -search_mode,
    +lift_mode;
```

Intended meaning:

1. stop searching;
2. start lifting.

But because intentions are interleaved, the agent may execute only the first deed:

```gwendolen
-search_mode
```

and then switch to another intention before executing:

```gwendolen
+lift_mode
```

At that moment, the belief base contains neither:

```gwendolen
search_mode
```

nor:

```gwendolen
lift_mode
```

Now the original achievement goal is reconsidered:

```gwendolen
+!holding(rubble) [achieve]
```

But neither plan applies:

First plan requires:

```gwendolen
B search_mode
```

Second plan requires:

```gwendolen
B lift_mode
```

So Gwendolen reports a warning/error:

```text
no applicable plan for goal holding(rubble)
```

The transcript explains this precise intermediate-state problem: search mode has been removed, but lift mode has not yet been added.

## 12.4 Why this problem happens

The two-deed sequence:

```gwendolen
-search_mode, +lift_mode
```

is not atomic by default.

After:

```gwendolen
-search_mode
```

the agent can select another intention.

This is dangerous when the belief base is being used to control program configuration. The agent can observe a half-updated configuration.

## 12.5 Locking syntax

The fix is to surround the mode update with lock/unlock deeds:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <-
    +.lock,
    -search_mode,
    +lift_mode,
    -.lock;
```

The full fixed program shown on the slides:

```gwendolen
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <-
        move_to(X, Y);

+!holding(rubble) [achieve] :
    {B rubble(X, Y), B at(X, Y), B lift_mode} <-
        lift_rubble;

+at(X, Y) :
    {~B rubble(X, Y)} <-
        +no_rubble(X, Y);

+rubble(X, Y) :
    {B at(X, Y)} <-
        +.lock,
        -search_mode,
        +lift_mode,
        -.lock;

+holding(rubble) :
    {True} <-
        print(done);
```

## 12.6 Meaning of lock

### Key concept: lock

**Intuition:**  
A lock tells Gwendolen to keep selecting the current intention until the lock is released.

**Formal mechanism from lecture:**  
Between:

```gwendolen
+.lock
```

and:

```gwendolen
-.lock
```

the current intention remains the current intention. Other intentions are not selected during that locked segment.

## 12.7 Worked trace: fixed mode switch

At the point where rubble is perceived at `(5, 5)`, the relevant event is:

```gwendolen
+rubble(5, 5)
```

The fixed plan expands into the deeds:

```gwendolen
+.lock
-search_mode
+lift_mode
-.lock
```

Execution:

1. Execute:

   ```gwendolen
   +.lock
   ```

   The current intention is now locked.

2. Execute:

   ```gwendolen
   -search_mode
   ```

   The belief `search_mode` is removed.

3. Because the intention is locked, Gwendolen does not switch to the `+!holding(rubble)` intention yet.

4. Execute:

   ```gwendolen
   +lift_mode
   ```

   The belief `lift_mode` is added.

5. Execute:

   ```gwendolen
   -.lock
   ```

   The lock is released.

6. Now when the `holding(rubble)` achievement goal is reconsidered, the lifting plan applies:

   ```gwendolen
   +!holding(rubble) [achieve] :
       {B rubble(X, Y), B at(X, Y), B lift_mode} <-
           lift_rubble;
   ```

7. The robot executes:

   ```gwendolen
   lift_rubble
   ```

8. Perception eventually adds:

   ```gwendolen
   holding(rubble)
   ```

9. The final plan prints:

   ```gwendolen
   print(done)
   ```

## 12.8 Locking advice

> **Exam flag — lecturer advice:** Be careful with lock/unlock. The lecturer’s advice is to never include an action inside a lock, especially an action that takes time to execute.

Reason:

- While locked, the agent cannot select other intentions.
- If a long-running action is inside a lock, the agent cannot react to other events during that action.
- Example from the lecture: while a robot is moving, you may want another intention checking for obstacles to run. If movement is inside a lock, that obstacle-checking intention cannot run.

### Good use of lock

Use lock for short sequences of internal configuration updates, such as:

```gwendolen
+.lock,
-search_mode,
+lift_mode,
-.lock
```

### Bad use of lock

Avoid:

```gwendolen
+.lock,
move_to(X, Y),
-.lock
```

because `move_to(X, Y)` may take time, and the agent should remain responsive while moving.

# 13. Waiting vs locking

## 13.1 Waiting

Use waiting when one intention must pause until some belief becomes true, but other intentions should keep running.

Example:

```gwendolen
move_to(X, Y),
*checked(X, Y)
```

Meaning:

- move to the square;
- do not continue this intention until the square has been checked;
- meanwhile, other intentions can process perceptions such as `+at(X, Y)` or `+rubble(X, Y)`.

## 13.2 Locking

Use locking when a short sequence of deeds must be executed without interleaving.

Example:

```gwendolen
+.lock,
-search_mode,
+lift_mode,
-.lock
```

Meaning:

- do not let another intention observe the intermediate state where neither mode is true.

## 13.3 Comparison

```text
Waiting:
    pauses current intention until condition is true;
    other intentions continue;
    good for waiting for perceptions/belief updates.

Locking:
    forces current intention to remain current;
    other intentions do not run during locked segment;
    good for short atomic belief/configuration updates.
```

> **Exam flag — contrast:** Waiting preserves responsiveness; locking temporarily reduces responsiveness.

## 14.2 Baseline rubble program with extra move

Actions:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
move_to(5, 5)
print(done)
```

Cause of extra move:

```text
lift_rubble creates a perception of holding(rubble)
but holding(rubble) is not yet in the belief base
so the achieve goal is reconsidered once more
and move_to(5, 5) is selected again
```

## 14.3 Reasoning-rule rubble program

Reasoning rule:

```gwendolen
square_to_check(X, Y) :-
    possible_rubble(X, Y),
    ~no_rubble(X, Y);
```

Plan:

```gwendolen
+!holding(rubble) [achieve] : {B square_to_check(X, Y)} <-
    move_to(X, Y);
```

Behaviour:

```text
Same behaviour as baseline program,
but the guard logic is factored into a reasoning rule.
```

## 14.4 List-recursive rubble program

Initial belief:

```gwendolen
possible_rubble([sq(1, 1), sq(3, 3), sq(5, 5)])
```

Rules:

```gwendolen
square_to_check(X, Y) :-
    possible_rubble(L),
    check_rubble(L, X, Y);

check_rubble([sq(X, Y) | T], X, Y) :-
    ~no_rubble(X, Y);

check_rubble([sq(X, Y) | T], X1, Y1) :-
    no_rubble(X, Y),
    check_rubble(T, X1, Y1);
```

Behaviour:

```text
Return first unchecked square in the list.
After no_rubble(1, 1), recurse to sq(3, 3).
After no_rubble(3, 3), recurse to sq(5, 5).
```

## 14.5 Goal-in-guard rubble relocation

Initial goal:

```gwendolen
rubble(2, 2) [achieve]
```

Plan:

```gwendolen
+!rubble(2, 2) [achieve] : {True} <-
    +!holding(rubble) [achieve],
    move_to(2, 2),
    drop;
```

Guarded lifting plan:

```gwendolen
+rubble(X, Y) :
    {B at(X, Y), G holding(rubble) [achieve]} <-
        lift_rubble;
```

Behaviour:

```text
Find rubble.
Lift it only if currently pursuing holding(rubble).
Move to (2, 2).
Drop it.
```

## 14.6 Waiting rubble program

Key plan:

```gwendolen
+!holding(rubble) [achieve] :
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <-
        move_to(X, Y),
        *checked(X, Y);
```

Effect:

```text
After moving to a square, wait until checked(X, Y) is believed.
This prevents premature repeated movement.
```

## 14.7 Locking mode-switch program

Buggy mode-switch:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <-
    -search_mode,
    +lift_mode;
```

Bug:

```text
Agent may switch intentions after -search_mode but before +lift_mode.
Then neither search plan nor lift plan applies.
```

Fixed mode-switch:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <-
    +.lock,
    -search_mode,
    +lift_mode,
    -.lock;
```

Effect:

```text
Mode update completes before another intention can run.
```

# 15. Exam flags and high-value points

## 15.1 Plans are event-triggered

**Know this:** A plan is triggered when a belief or goal is added or removed, not merely because a condition is true.

The obstacle example is the key warning:

```gwendolen
+obstacle : {T} <- turn_90, forward.
```

The robot does not repeatedly turn because `obstacle` stays true rather than appearing again.

## 15.2 Guards express current conditions

Triggers are for events. Guards are for conditions that must currently hold.

Example:

```gwendolen
+at(X, Y) : {~B rubble(X, Y)} <- +no_rubble(X, Y);
```

The event is:

```gwendolen
+at(X, Y)
```

The condition is:

```gwendolen
~B rubble(X, Y)
```

## 15.3 Achieve vs perform goals

Perform goals:

```text
plan, execute, drop
```

Achieve goals:

```text
check belief base;
if not achieved, plan and execute;
check again;
if still not achieved, plan again
```

## 15.4 Intentions are stacks of tuples

Know the tuple structure:

```text
(trigger event, deed, unifier)
```

Know the special deeds:

```text
no plan yet
null
```

## 15.5 `no plan yet` means planning has not happened

When the top deed is:

```text
no plan yet
```

Gwendolen searches for applicable plans.

## 15.6 Perception creates intentions

Perceived information is not instantly a belief. It creates an intention to add the belief.

This is why the rubble program can lift rubble but still not yet believe:

```gwendolen
holding(rubble)
```

## 15.7 The extra move is explainable

The extra:

```gwendolen
move_to(5, 5)
```

comes from the achieve-goal intention being selected again before the `holding(rubble)` belief has been added.

## 15.8 Reasoning rules can simplify guards

Instead of putting everything in the plan guard:

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

define:

```gwendolen
square_to_check(X, Y) :- possible_rubble(X, Y), ~no_rubble(X, Y);
```

and use:

```gwendolen
{B square_to_check(X, Y)}
```

## 15.9 Lists and recursion are allowed in reasoning rules

The list version uses:

```gwendolen
[sq(X, Y) | T]
```

and recursive `check_rubble`.

## 15.10 Goals can appear in guards

Example:

```gwendolen
G holding(rubble) [achieve]
```

This checks whether the agent currently has that goal.

## 15.11 Waiting controls progress of one intention

Wait syntax:

```gwendolen
*checked(X, Y)
```

Meaning:

```text
Do not continue this intention until checked(X, Y) is believed.
```

## 15.12 Locking prevents interleaving

Lock syntax:

```gwendolen
+.lock
...
-.lock
```

Meaning:

```text
Keep the current intention current until the lock is released.
```

## 15.13 Locking warning

**Lecturer advice:** Do not put long-running actions inside locks. In particular, do not put actions such as movement inside a lock if the robot should remain able to react to obstacles or other events.

# 16. Connections across lectures

## 16.1 Intro lecture → events/intentions lecture

The intro lecture introduces:

- Gwendolen as a BDI language;
- beliefs, goals, plans;
- achieve vs perform goals;
- the reasoning cycle;
- the rubble example and the extra move.

The events/intentions lecture then explains the machinery behind that extra move:

- events;
- start events;
- intentions as stacks;
- `no plan yet`;
- unifiers;
- perception creating intentions.

## 16.2 Events/intentions lecture → waiting/locking lecture

The events lecture explains why intentions can interleave in surprising ways. The waiting/locking lecture gives mechanisms to control that interleaving:

- waiting fixes premature continuation of an intention;
- locking fixes interruption during a short critical update.

## 16.3 Plan guards lecture → previous Datalog/Prolog material

The lecturer explicitly connects reasoning rules to Datalog/Prolog-like rules and unification. The recursive list example relies on the same kind of unification used in logic programming.

## 16.4 Gwendolen → Jason

The lecturer says Jason is more widely used, but many concepts in Gwendolen also appear in Jason. Learning Gwendolen should transfer to other BDI languages.

## 16.5 Gwendolen → verification

The lecturer motivates Gwendolen partly by its application to verification of agent programs, which is flagged as material for the following week.

## 16.6 Agent plans → low-level robotics

Actions such as:

```gwendolen
move_to(X, Y)
lift_rubble
pick-up(block)
```

are high-level calls. The low-level control is delegated to the environment or robotics code.
