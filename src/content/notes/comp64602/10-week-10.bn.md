# 1. বড় ছবি: Gwendolen এবং BDI programming

## 1.1 Gwendolen কী

Gwendolen একটি **Beliefs–Desires–Intentions** programming language, যা agent programming-এর জন্য ব্যবহৃত হয়। লেকচারার এটিকে **Jason**-এর সঙ্গে তুলনা করেছেন। Jason বেশি প্রচলিত, কিন্তু এই কোর্সে Gwendolen ব্যবহার করা হচ্ছে কারণ এটি লেকচারার নিজে তৈরি করেছেন এবং agent program-এর **verification**-এ এর বিশেষ ব্যবহার আছে, যা কোর্সে পরে দেখা হবে।

### মূল ধারণা: BDI programming

**সহজ ধারণা:**  
একটি BDI agent-কে program করা হয় সে পৃথিবী সম্পর্কে কী বিশ্বাস করে, তার কী goal আছে, এবং সে এখন কোন কোন active line of action অনুসরণ করছে — এগুলোর ভিত্তিতে।

**এই লেকচারের Gwendolen setting-এ:**

- **Beliefs** হলো agent-এর belief base-এ থাকা facts, যেমন `empty`, `at(5, 5)`, বা `rubble(5, 5)`।
- **Goals** হলো agent যা ঘটাতে বা করতে চায়, যেমন `pickup [achieve]` বা `holding(rubble) [achieve]`।
- **Intentions** হলো agent বর্তমানে যেসব active কাজের stack process করছে।
- **Plans** বলে দেয় কোনো event ঘটলে নির্দিষ্ট condition-এর অধীনে agent কীভাবে react করবে।

## 1.2 BDI/Gwendolen কেন “high-level”

Gwendolen-কে **high-level** language হিসেবে ভাবা হয়: এটি agent-এর বড় decision-making নিয়ে কাজ করে, low-level control নয়। যেমন কোনো plan-এ action থাকতে পারে:

```gwendolen
pick-up(block)
```

অথবা:

```gwendolen
lift_rubble
```

Gwendolen program মোটর চালানো, gripper নিয়ন্ত্রণ, force detect করা, বা robotics-এর খুঁটিনাটি বলে না। সেগুলো environment বা lower-level control code সামলাবে বলে ধরে নেওয়া হয়।

### মূল ধারণা: action

**সংজ্ঞা:**  
একটি **action** হলো plan-এর এমন deed যার সামনে `+`, `-`, বা `!` decoration নেই। এটি এমন কিছু যা agent environment বা low-level system-কে করতে বলে।

উদাহরণ:

```gwendolen
pick-up(block)
move_to(X, Y)
lift_rubble
print(done)
```

Action-কে এমন deeds থেকে আলাদা করতে হবে যেগুলো belief/goal add বা remove করে, যেমন:

```gwendolen
+pickup
-empty
+!holding(rubble) [achieve]
```

## 1.3 BDI/Gwendolen কেন “declarative”

Language-এর উদ্দেশ্য হলো solution-এর প্রতিটি detail লিখে দেওয়ার বদলে problem-টি express করা। Programmer plans লেখে beliefs, goals, এবং conditions-এর ভাষায়।

Intro slides-এর উদাহরণ:

```gwendolen
+!empty : {B holding(block)} <- 
    send(co-worker, take(co-worker, block));

+!empty : {B holding(A)} <- 
    send(co-worker, take(co-worker, A));

+!holding(block) : {B empty, B on_ground(block)} <- 
    pick-up(block);

+on_ground(block) : {~G holding(block)} <- 
    +!holding(block);
```

ব্যাখ্যা:

- Agent যদি empty hands-এর goal পায় এবং বিশ্বাস করে যে সে block ধরে আছে, তাহলে co-worker-কে message পাঠায় যেন co-worker সেটি নিয়ে যায়।
- Agent যদি block ধরে রাখার goal পায় এবং বিশ্বাস করে যে তার hands empty এবং block ground-এ আছে, তাহলে `pick-up(block)` action করে।
- Agent যদি perceive করে যে block ground-এ আছে এবং তার কাছে block ধরার goal না থাকে, তাহলে সে block ধরার goal adopt করে।

## 1.4 “Declarative … অথবা সবসময় নয়”

লেকচারার সতর্ক করেছেন যে বাস্তব program অনেক সময় পরিষ্কার ও ছোট থাকে না। Satellite-control example-এ একই event-এর জন্য একাধিক plan আছে:

```gwendolen
+!maintaining(Pos) [achieve] : {B in_position(Pos), B assuming_formation(F), 
    ~B aborted(Reason), ~B broken(X)} <- 
        perf(maintain_path), 
        +maintaining(Pos), 
        +!cleanup [perform];

+!maintaining(Pos) [achieve] : {~B in_position(Pos), B assuming_formation(F), 
    ~B aborted(Reason), ~B broken(X)} <- 
        +!in_position(Pos) [achieve], 
        perf(maintain_path), 
        +maintaining(Pos), 
        +!cleanup [perform];

+!maintaining(Pos) [achieve] : {B broken(X), ~B aborted(Reason)} <- 
        *fixed(X), 
        -fixed(X);

+!maintaining(Pos) [achieve] : {~B assuming_formation(F)} <- 
        -!maintaining(Pos) [achieve];

+!maintaining(Pos) [achieve] : {B aborted(Reason), B assuming_formation(F)} <- 
        -!assuming_formation(F) [perform];
```

এখানে satellite-control details মুখ্য নয়। মূল point হলো: বাস্তব agent program-এ একই goal-এর জন্য অনেক plan লাগতে পারে, belief অনুযায়ী কোন plan লাগবে তা নির্ভর করে। যেমন:

- agent ইতিমধ্যে position-এ আছে কি না;
- formation assume করছে কি না;
- mission abort হয়েছে কি না;
- কোনো component broken কি না।

**Exam flag — গুরুত্বপূর্ণ সতর্কতা:** BDI language declarative হতে চায়, কিন্তু non-trivial application-এ অনেক plan এবং complicated guard লাগতে পারে।

# 2. Basic Gwendolen program structure

একটি Gwendolen program-এ সাধারণত এমন sections থাকে:

```gwendolen
GWENDOLEN

:name: ag1

:Initial Beliefs:
empty

:Initial Goals:
pickup [achieve]

:Plans:
+!pickup [achieve] : {B empty} <- 
    +pickup, 
    -empty, 
    print(done);
```

এটি slides-এর simple pick-up example।

## 2.1 Program-এর অংশগুলো

### `:name:`

Agent-এর নাম দেয়।

উদাহরণ:

```gwendolen
:name: ag1
```

লেকচারার বলেছেন, একটি program-এ একাধিক agent থাকতে পারে, তাই agent-দের নাম দরকার।

### `:Initial Beliefs:`

Agent-এর belief base-এ শুরুতে থাকা facts।

উদাহরণ:

```gwendolen
empty
```

Agent শুরুতে বিশ্বাস করে যে তার hands empty।

### `:Initial Goals:`

Agent-কে শুরুতে দেওয়া goals।

উদাহরণ:

```gwendolen
pickup [achieve]
```

Agent `pickup` achieve করার goal নিয়ে শুরু করে।

### `:Plans:`

Events-এর প্রতি কীভাবে react করতে হবে তা বলা rules।

সাধারণ form:

```gwendolen
trigger : {guard} <- deed1, deed2, ..., deedN;
```

উদাহরণ:

```gwendolen
+!pickup [achieve] : {B empty} <- 
    +pickup, 
    -empty, 
    print(done);
```

ব্যাখ্যা:

- Trigger: `pickup [achieve]` goal এসেছে।
- Guard: agent `empty` বিশ্বাস করে।
- Deeds:
  - belief `pickup` add করা;
  - belief `empty` remove করা;
  - `done` print করা।

## 2.2 Plan syntax

### Trigger event

Colon-এর আগে থাকা অংশ:

```gwendolen
+!pickup [achieve]
```

অর্থ:

- `+` মানে কিছু appeared/added হয়েছে।
- `!` মানে সেটি goal।
- `!` না থাকলে সেটি belief।
- `-` মানে কিছু disappeared/removed হয়েছে।

উদাহরণ:

```gwendolen
+!holding(rubble) [achieve]
```

`holding(rubble)` achieve করার goal এসেছে।

```gwendolen
+at(X, Y)
```

`at(X, Y)` belief এসেছে।

```gwendolen
-rubble(X, Y)
```

`rubble(X, Y)` belief disappeared/removed হয়েছে।

### Guard

Curly braces-এর ভিতরের condition:

```gwendolen
{B empty}
```

অর্থ: plan apply হবে কেবল যদি agent `empty` বিশ্বাস করে।

লেকচারগুলোতে ব্যবহৃত common guard notation:

```gwendolen
B p
```

Agent `p` বিশ্বাস করে।

```gwendolen
~B p
```

Agent `p` বিশ্বাস করে না।

```gwendolen
G g [achieve]
```

Agent-এর goal `g [achieve]` আছে।

```gwendolen
True
```

Guard সবসময় true।

### Deeds

`<-`-এর ডান পাশে থাকা instructions।

উদাহরণ:

```gwendolen
+pickup
-empty
print(done)
move_to(X, Y)
+!holding(rubble) [achieve]
```

Deed করতে পারে:

- belief add করা;
- belief remove করা;
- goal add করা;
- goal remove করা;
- action call করা;
- belief-এর জন্য wait করা;
- intention lock/unlock করা।

# 3. Achieve goals বনাম perform goals

লেকচারগুলোতে **achieve goals** এবং **perform goals** আলাদা করা হয়েছে। এটি একটি major concept এবং introductory lecture ও events/intentions lecture — দুটোতেই এসেছে।

## 3.1 Perform goals

### সংজ্ঞা

Agent যখন একটি **perform goal** add করে, তখন সে:

1. ওই goal-এর জন্য plan খোঁজে;
2. plan execute করে;
3. goal drop করে।

Perform goal সাধারণত code structure করার জন্য ব্যবহৃত হয়, subroutine বা method-এর মতো।

উদাহরণ:

```gwendolen
+!goto(Room) [perform] : {True} <- 
    planRoute(Room, Route), 
    follow(Route), 
    enter(Room);
```

এই goal কোনো belief true হওয়া পর্যন্ত maintained থাকে না। Agent শুধু procedure perform করে এবং goal drop করে।

### সহজ ধারণা

Perform goal মানে: “এই procedure-টা করো।”

এর মানে নয়: “world এই condition satisfy করা পর্যন্ত keep trying করো।”

## 3.2 Achieve goals

### সংজ্ঞা

Agent যখন একটি **achieve goal** add করে, তখন সে:

1. belief base দেখে goal ইতিমধ্যে achieved কি না check করে;
2. achieved না হলে goal-এর জন্য plan খুঁজে execute করে;
3. plan execute করার পরে আবার check করে corresponding belief আছে কি না;
4. belief না থাকলে আবার plan search করে।

উদাহরণ:

```gwendolen
holding(rubble) [achieve]
```

এই goal active থাকে যতক্ষণ agent বিশ্বাস না করে:

```gwendolen
holding(rubble)
```

### সহজ ধারণা

Achieve goal মানে: “এটা true করো, এবং আমি এটা true বিশ্বাস করা পর্যন্ত চেষ্টা চালিয়ে যাও।”

## 3.3 Worked example: simple `pickup [achieve]`

Program:

```gwendolen
GWENDOLEN

:name: ag1

:Initial Beliefs:
empty

:Initial Goals:
pickup [achieve]

:Plans:
+!pickup [achieve] : {B empty} <- 
    +pickup, 
    -empty, 
    print(done);
```

Execution:

1. Agent শুরু করে initial belief দিয়ে:

   ```gwendolen
   empty
   ```

2. Initial goal:

   ```gwendolen
   pickup [achieve]
   ```

3. এটি initial goal হওয়ায় Gwendolen goal add করার জন্য start event তৈরি করে।

4. `+!pickup [achieve]` plan apply হয় কারণ guard true:

   ```gwendolen
   {B empty}
   ```

5. Plan execute হয়:

   ```gwendolen
   +pickup
   -empty
   print(done)
   ```

6. Belief base পরিবর্তিত হয়:

   ```gwendolen
   empty
   ```

   থেকে:

   ```gwendolen
   pickup
   ```

7. Program print করে:

   ```text
   done
   ```

লেকচারার বলেছেন এটি বাস্তব robotics নয়: robot শুধু belief add করে কোনো জিনিস তুলে ফেলতে পারে না। এটি mechanics বোঝানোর ছোট example।

# 4. Gwendolen reasoning cycle

Reasoning-cycle diagram intro slides এবং events/intentions slides — দুই জায়গাতেই এসেছে। Transcript-এ stages-এর order-এর ওপর জোর দেওয়া হয়েছে, কারণ এই order-ই rubble example-এ “extra move” ব্যাখ্যা করে।

## 4.1 Algorithm: Gwendolen reasoning cycle

Cycle:

1. **Stage A — current intention select করা অথবা agent-কে sleep করা।**
   - Gwendolen একটি intention বেছে নেয়।
   - Defaultভাবে এটি intentions-গুলোর মধ্যে cycle করে।
   - usable intention না থাকলে agent sleep করতে পারে বা perception stage-এ যেতে পারে।

2. **Decision — unsuspended non-empty intention আছে কি?**
   - থাকলে planning-এ যায়।
   - না থাকলে perception-এ যায়।

3. **Stage B — current intention-এর applicable plans খোঁজা।**
   - Current intention-এর top deed যদি `no plan yet` হয়, সাধারণত তখন planning লাগে।

4. **Stage C — একটি plan pick করে apply করা।**
   - একাধিক applicable plan থাকলে একটি select হয়।
   - Plan apply করলে সাধারণত plan-এর deeds intention stack-এর ওপরে যোগ হয়।

5. **Decision — current intention empty কি?**
   - empty হলে perception-এ যায়।
   - না হলে top deed execute করে।

6. **Stage D — current intention-এর top deed execute করা।**
   - এটি action, belief addition/removal, goal addition/removal, wait, lock ইত্যাদি হতে পারে।

7. **Stage E — নতুন perception ও messages নেওয়া।**
   - Perceptions নতুন intentions তৈরি করতে পারে।
   - যেমন `at(1, 1)` perceive করলে `at(1, 1)` belief add করার intention তৈরি হয়।

8. **Stage F — নতুন messages process করা।**
   - Messages-ও events/intentions তৈরি করতে পারে।

9. Stage A-তে ফিরে যাওয়া।

## 4.2 Order কেন গুরুত্বপূর্ণ

মূল order:

```text
select intention
→ plan if needed
→ execute top deed
→ get perception/messages
→ select another intention
```

এই order-এর ফলে:

- Actions নতুন perceptions ঘটাতে পারে।
- Perceptions সঙ্গে সঙ্গে beliefs হয় না।
- Perceptions প্রথমে beliefs add করার intentions তৈরি করে।
- সেই intention process হলেই belief add হয়।

তাই action কোনো percept তৈরি করলেও corresponding belief belief base-এ ঢুকতে একটু lag হয়।

**Exam flag — high value:** Perception lag-ই rubble example-এ extra `move_to(5, 5)`-এর কেন্দ্রীয় কারণ।

# 5. Events

## 5.1 Event-এর সংজ্ঞা

Plans **events** দ্বারা triggered হয়। Event হলো কোনো পরিবর্তন: belief বা goal add হওয়া বা remove হওয়া।

Formal examples:

```gwendolen
+belief
-belief
+!goal [achieve]
-!goal [achieve]
```

কোনো condition true বলেই plan fire করে না। Triggering event ঘটতে হবে এবং guard satisfy করতে হবে।

## 5.2 Special `start` event

একটি special `start` event আছে, যা ব্যবহৃত হয়:

1. agent-এর initial goals-এর জন্য;
2. perception থেকে আসা information belief-এ convert করার জন্য।

Example initial goal intention:

```text
Trigger: +start
Deed:    +!holding(rubble)
```

Example perceptual intention:

```text
Trigger: +start
Deed:    +at(1, 1)
```

`start` event ব্যবহৃত হয় কারণ initial goal বা perceptual belief-কে agent-এর belief/goal machinery-তে ঢোকাতে হয়।

## 5.3 Common mistake: events আর conditions গুলিয়ে ফেলা

Obstacle example:

```gwendolen
+obstacle : {T} <- turn_90, forward.
```

Scenario:

1. Robot একটি corner-এর দিকে এগোয়।
2. Obstacle detect করে।
3. `obstacle` belief appears।
4. `+obstacle` event plan trigger করে।
5. Robot 90 degrees turn করে।
6. Robot forward যায়।
7. Robot অন্য wall-এ hit করে।

কেন আবার turn করে না?

কারণ robot turn করার সময় obstacle belief disappear করে আবার reappear করেনি। Belief একইভাবে true ছিল। তাই নতুন `+obstacle` event ঘটেনি, ফলে plan আবার trigger হয়নি।

**Exam flag — common mistake:** Plans changes/events দ্বারা triggered হয়; facts merely true থাকলেই plan trigger হয় না। True facts guard-এ ব্যবহার করো; সদ্য changed/appeared বিষয় trigger-এ ব্যবহার করো।

# 6. Intentions

## 6.1 Intention-এর সংজ্ঞা

একটি **intention** হলো tuples-এর stack। প্রতিটি tuple-এ থাকে:

```text
(trigger event, deed, unifier)
```

Slides-এ column:

```text
Trigger | Deed | Unifier
```

লেকচারার পরের examples-এ অনেক সময় unifier column বাদ দিয়ে simplified stack দেখিয়েছেন, কিন্তু underlying structure-এ unifier থাকে।

### মূল ধারণা: unifier

**সহজ ধারণা:**  
Unifier plan/event/guard matching-এর সময় তৈরি variable bindings record করে।

উদাহরণ:

```text
{Room = room1}
```

মানে variable `Room` concrete value `room1`-এর সঙ্গে unified হয়েছে।

### মূল ধারণা: deed

একটি **deed** হলো tuple-এর current instruction। এটি action, belief update, goal update, `no plan yet`, `null`, wait, lock ইত্যাদি হতে পারে।

## 6.2 Special deed: `no plan yet`

Slides-এ abbreviation:

```text
npy
```

Transcript-এ কখনও “NPI” বা “no plan yet” এসেছে।

সংজ্ঞা:

```text
no plan yet
```

মানে agent-এর intention stack-এ event আছে, কিন্তু সেটি কীভাবে handle করবে তা এখনও decide করেনি। Top deed যদি `no plan yet` হয়, Gwendolen applicable plans খোঁজে।

## 6.3 Special deed: `null`

Perform goals-এর ক্ষেত্রে perform goal add করার original deed `null` দিয়ে replace হয়।

অর্থ:

- Perform goal একটি নতুন event হয়ে planning-এ যায়।
- Original deed achievement checking-এর জন্য রেখে দেওয়া হয় না।
- Perform goal-এর plan execute হয়ে গেলে goal disappear করে।

এটি achieve goal থেকে আলাদা। Achieve goal-এ goal deed থেকে যায়, যাতে Gwendolen পরে check করতে পারে goal সত্যিই achieved হয়েছে কি না।

# 7. Worked example: `clean` intention stack

Events lecture-এ এই program ব্যবহার করা হয়েছে:

```gwendolen
GWENDOLEN

:name: temp

:Initial Beliefs:

:Initial Goals:
clean [achieve]

:Plans:
+!clean [achieve] : {B dirty(Room)} <- 
    +!goto(Room) [perform], 
    +!vacuum(Room) [perform];

+!goto(Room) [perform] : {T} <- 
    planRoute(Room, Route), 
    follow(Route), 
    enter(Room);
```

ধরা যাক agent বিশ্বাস করে:

```gwendolen
dirty(room1)
```

## 7.1 Initial intention

Initial goals `start` event ব্যবহার করে।

```text
Trigger     Deed                  Unifier
+start      +!clean [achieve]      {}
```

Top deed হলো achieve goal `clean` add করা।

যেহেতু `clean` একটি achieve goal, agent আগে check করে সে ইতিমধ্যে বিশ্বাস করে কি না:

```gwendolen
clean
```

সে বিশ্বাস করে না, তাই goal add হয়।

## 7.2 Achieve goal add করলে নতুন event তৈরি হয়

Stack becomes:

```text
Trigger              Deed             Unifier
+!clean [achieve]    no plan yet       {}
+start               +!clean [achieve] {}
```

Top row-এর অর্থ:

- event হলো `clean [achieve]` goal acquire করা;
- agent এখনো এই event-এর জন্য plan choose করেনি।

## 7.3 `clean` plan apply করা

Plan:

```gwendolen
+!clean [achieve] : {B dirty(Room)} <- 
    +!goto(Room) [perform], 
    +!vacuum(Room) [perform];
```

apply হয় কারণ agent বিশ্বাস করে:

```gwendolen
dirty(room1)
```

তাই unification gives:

```text
Room = room1
```

`no plan yet` remove হয় এবং plan deeds intention-এর ওপরে বসে:

```text
Trigger              Deed                         Unifier
+!clean [achieve]    +!goto(Room) [perform]        {Room = room1}
+!clean [achieve]    +!vacuum(Room) [perform]      {Room = room1}
+start               +!clean [achieve]             {}
```

## 7.4 Perform goal handle করা

Perform goal `+!goto(Room) [perform]` event হলে original deed `null` দিয়ে replace হয়।

```text
Trigger                  Deed           Unifier
+!goto(Room) [perform]   no plan yet     {Room = room1}
+!clean [achieve]        null            {Room = room1}
+!clean [achieve]        +!vacuum(Room)  {Room = room1}
+start                   +!clean         {}
```

ব্যাখ্যা:

- Agent এখন `goto(room1)` perform করার plan খুঁজবে।
- Original perform-goal deed `null` হয়ে গেছে।
- `goto` plan-এর deeds execute হলে system stack-এর নিচের দিকে continue করবে।
- Perform goal achievement re-check করার জন্য থাকে না।

## 7.5 `goto` plan apply করা

Plan:

```gwendolen
+!goto(Room) [perform] : {T} <- 
    planRoute(Room, Route), 
    follow(Route), 
    enter(Room);
```

Guard `{T}` হওয়ায় plan সবসময় applicable।

Deeds:

```gwendolen
planRoute(Room, Route)
follow(Route)
enter(Room)
```

এসব actions। Execute হওয়ার সঙ্গে সঙ্গে intention থেকে remove হয়, যদি না তারা নতুন events তৈরি করে।

# 8. Multiple intentions

## 8.1 কী intention শুরু করে?

প্রতিটি event যেটির প্রতি react করা হচ্ছে একটি intention শুরু করে। এর মধ্যে আছে:

- initial goal acquire করা;
- perception থেকে information নিয়ে belief করা;
- belief add করা;
- belief remove করা;
- goal add/remove করা।

Slides-এ বলা হয়েছে, initial goal বা belief acquire করার মতো প্রতিটি event একটি intention শুরু করে। Gwendolen intentions-গুলোর মধ্যে cycle করে এবং প্রত্যেকটির top deed turn অনুযায়ী handle করে।

## 8.2 Multiple intentions কেন useful

এগুলো agent-কে existing goals pursue করার সময় নতুন events-এর প্রতি react করতে দেয়।

Example intuition:

- Robot rubble-clearing goal pursue করছে।
- নতুন obstacle perceive হতে পারে।
- Obstacle event নিজস্ব intention start করতে পারে।
- Agent original goal abandon না করেই obstacle react করতে পারে।

## 8.3 Multiple intentions কেন problem তৈরি করে

Intentions interleaved হওয়ায় agent একটি partially completed sequence থেকে অন্য intention-এ switch করতে পারে।

এটি responsiveness-এর জন্য ভালো, কিন্তু খারাপ যখন:

- belief updates-এর sequence একসঙ্গে হওয়া দরকার;
- current intention-কে perceptual processing-এর জন্য অপেক্ষা করতে হবে;
- mode switch অর্ধেক completed;
- একটি intention belief base-কে intermediate state-এ দেখে।

এই lecture cluster শেষে দুইটি mechanism আসে:

```text
waiting
locking
```

# 9. Baseline worked example: pick up rubble

একই rubble program বারবার lectures-এ ব্যবহার করা হয়েছে।

## 9.1 Program

```gwendolen
GWENDOLEN

:name: robot

:Initial Beliefs:
possible_rubble(1, 1)
possible_rubble(3, 3)
possible_rubble(5, 5)

:Initial Goals:
holding(rubble) [achieve]

:Plans:
+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <- 
        move_to(X, Y);

+at(X, Y) : 
    {~B rubble(X, Y)} <- 
        +no_rubble(X, Y);

+rubble(X, Y) : 
    {B at(X, Y)} <- 
        lift_rubble;

+holding(rubble) : 
    {True} <- 
        print(done);
```

## 9.2 Intended environment

Lecture-এর environment:

```text
No rubble at (1, 1)
No rubble at (3, 3)
Rubble at (5, 5)
```

Natural expected behaviour:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
print(done)
```

কিন্তু actual behaviour-এ extra move থাকে:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
move_to(5, 5)
print(done)
```

Extra `move_to(5, 5)` ঘটে achieve goals, perception lag, এবং multiple intentions-এর interaction-এর কারণে।

## 9.3 প্রথম plan কেন বারবার apply হয়

Initial goal:

```gwendolen
holding(rubble) [achieve]
```

Agent যতক্ষণ বিশ্বাস না করে ততক্ষণ goal active থাকে:

```gwendolen
holding(rubble)
```

Goal achieve করার plan:

```gwendolen
+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <- 
        move_to(X, Y);
```

Achievement goal reconsidered হলে agent এমন square খোঁজে যেখানে:

```gwendolen
possible_rubble(X, Y)
```

believed, এবং:

```gwendolen
no_rubble(X, Y)
```

believed নয়।

যদি `(1, 1)` already ruled out হয় `no_rubble(1, 1)` add করে, তাহলে next applicable square `(3, 3)`। `(3, 3)` ruled out হলে next square `(5, 5)`।

Initial intention:

```text

+start      +!holding(rubble)
```

Initial goal add হয়। Achieve goal হওয়ায় এবং agent `holding(rubble)` বিশ্বাস না করায় Gwendolen নতুন goal event তৈরি করে:

```text
Trigger                  Deed
+!holding(rubble)        no plan yet
+start                   +!holding(rubble)
```

First plan applies with:

```text
X = 1
Y = 1
```

কারণ:

```gwendolen
possible_rubble(1, 1)
```

believed এবং:

```gwendolen
no_rubble(1, 1)
```

believed নয়।

Intention becomes:

```text
Trigger                  Deed
+!holding(rubble)        move_to(1, 1)
+start                   +!holding(rubble)
```

Action executes:

```gwendolen
move_to(1, 1)
```

তারপর perception হয়।

## 9.5 `(1, 1)`-এ move করার পর perception

Move করার পর agent perceive করে যে সে `(1, 1)`-এ আছে। Perception একটি intention তৈরি করে:


apply হয়, কারণ `(1, 1)`-এ agent rubble বিশ্বাস করে না।

Achieve goal `holding(rubble)` এখনও satisfied নয়, কারণ agent বিশ্বাস করে না:


Achieve goal still unsatisfied। `(1, 1)` এবং `(3, 3)` ruled out। Next possible square `(5, 5)`।

Event:

```gwendolen
+rubble(5, 5)
```

এই plan trigger করে:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- lift_rubble;
```

Guard holds কারণ agent বিশ্বাস করে:

```gwendolen

```

তাই action executes:


```text

+start      +holding(rubble)
```

## 9.9 Extra `move_to(5, 5)` কেন হয়

নিচের belief add করার intention process হওয়ার আগে:

```gwendolen
holding(rubble)
```

Gwendolen original achieve-goal intention-এ ফিরে যেতে পারে:

```gwendolen
+!holding(rubble) [achieve]
```

সে মুহূর্তে:

- agent `lift_rubble` করেছে;
- environment perception দিয়েছে যে সে rubble ধরে আছে;
- কিন্তু belief `holding(rubble)` এখনো belief base-এ নেই।

তাই achieve goal এখনো unsatisfied ধরা হয়।

First plan still applies for `(5, 5)` কারণ:

```gwendolen
possible_rubble(5, 5)
```

believed, এবং:

```gwendolen
no_rubble(5, 5)
```

believed নয়।

So agent executes:

```gwendolen

```

আবার।

তারপর perception intention process হয় এবং add করে:

```gwendolen
holding(rubble)
```

এখন `+holding(rubble)` event trigger করে:

```gwendolen

```

Agent prints:

```text
done
```

# 10. Plan guards and reasoning rules

এই lecture rubble example-কে extend করে: কিছু guard logic reasoning rules-এ সরানো হয়, তারপর lists, recursion, এবং goals in guards দেখানো হয়।

## 10.1 Plan guards

একটি **plan guard** হলো `{ ... }`-এর ভিতরের condition, যা hold করলে plan apply হয়।

উদাহরণ:

```gwendolen
+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <- 
        move_to(X, Y);
```

Guard says:

- agent বিশ্বাস করে `(X, Y)`-তে rubble থাকতে পারে;
- agent বিশ্বাস করে না যে `(X, Y)`-তে rubble নেই।

### মূল ধারণা: guard বনাম trigger

**Trigger:** সদ্য কী ঘটেছে।

```gwendolen
+!holding(rubble) [achieve]
```

**Guard:** বর্তমানে কী true হতে হবে।

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

Trigger event-based; guard state-based।

## 10.2 Reasoning rules

Reasoning rules guard-এর ভিতরে beliefs থেকে useful predicates derive করতে ব্যবহৃত হয়।

### Formal syntax

```gwendolen
Head :- Body;
```

Transcript বলে এটি Prolog-like এবং Datalog-এর মতো। `:-` implication-এর মতো arrow: body hold করলে head holds।

## 10.3 Rubble program-এর reasoning-rule version

Slides-এর program:

```gwendolen
GWENDOLEN

:name: robot

:Initial Beliefs:
possible_rubble(1, 1)
possible_rubble(3, 3)
possible_rubble(5, 5)

:Reasoning Rules:
square_to_check(X, Y) :- 
    possible_rubble(X, Y), 
    ~no_rubble(X, Y);

:Initial Goals:
holding(rubble) [achieve]

:Plans:
+!holding(rubble) [achieve] : 
    {B square_to_check(X, Y)} <- 
        move_to(X, Y);

+at(X, Y) : 
    {~B rubble(X, Y)} <- 
        +no_rubble(X, Y);

+rubble(X, Y) : 
    {B at(X, Y)} <- 
        lift_rubble;

+holding(rubble) : 
    {True} <- 
        print(done);
```

## 10.4 `square_to_check`-এর অর্থ

Reasoning rule:

```gwendolen
square_to_check(X, Y) :- 
    possible_rubble(X, Y), 
    ~no_rubble(X, Y);
```

### সহজ ধারণা

একটি square check করার মতো যদি:

1. সেখানে rubble থাকতে পারে; এবং
2. আমরা আগে থেকে establish করিনি যে সেখানে rubble নেই।

### Formal reading

যে কোনো `X, Y`-এর জন্য:

```text
square_to_check(X, Y)
```

holds if:

```text
possible_rubble(X, Y)
```

holds and:

```text
no_rubble(X, Y)
```

does not hold।

এতে plan guard clean হয়:

```gwendolen
{B square_to_check(X, Y)}
```

instead of:

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

Behaviour previous version-এর মতোই।

## 10.5 Guards-এ lists ব্যবহার

Next version তিনটি আলাদা belief-এর বদলে possible rubble locations এক list-এ রাখে।

Initial belief:


```gwendolen

    check_rubble(T, X1, Y1);
```


এই case-এ:


## 10.10 Goals in guards

Lecture দেখায় যে goals guard-এর মধ্যে থাকতে পারে।

Key plan:


# 11. Multiple intentions control করা: waiting

পরের lecture-এ multiple intentions control করার দুই mechanism এসেছে: **waiting** এবং **locking**। Waiting প্রথমে ব্যবহার করা হয়েছে যেন rubble-searching agent current square process করার আগে আরেক movement-এ “charge ahead” না করে।

## 11.1 যে problem solve হচ্ছে

Baseline rubble program-এ achieve goal `holding(rubble)` বারবার reconsidered হয়। Perceptions ও belief updates separate intentions দিয়ে process হওয়ায় agent previous action-এর perceptual consequences fully processed হওয়ার আগে original goal intention আবার choose করতে পারে।

Waiting programmer-কে বলতে দেয়:

```text
এই intention continue করো না যতক্ষণ না একটি particular belief true হয়।
```

## 11.2 Wait syntax

Wait deed `*` দিয়ে লেখা হয়।

Example:

```gwendolen
*checked(X, Y)
```

Meaning:

```text
Agent checked(X, Y) বিশ্বাস করা পর্যন্ত wait করো।
```

Transcript says “star means wait for.”

### Important detail

Waiting ওই intention-কে suspend/skip করে যতক্ষণ waited-for belief appear না করে। এটি পুরো agent বন্ধ করে না। অন্য intentions চলতে পারে।

## 11.3 Waiting version of the rubble program

Slides-এর waiting version:

```gwendolen
:Plans:

+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <- 
        move_to(X, Y),
        *checked(X, Y);

+at(X, Y) : 
    {~B rubble(X, Y)} <- 
        +no_rubble(X, Y), 
        +checked(X,Y);

+rubble(X, Y) : 
    {B at(X, Y)} <- 
        lift_rubble, 
        +checked(X, Y);

+holding(rubble) : 
    {True} <- 
        print(done);
```

[UNCLEAR] প্রথম waiting slide-এ guard-এ `B search_mode` আছে, কিন্তু subsequent worked waiting trace slides মূলত wait/check mechanism-এ focus করে এবং অনেক সময় simpler guard দেখায়। Wait mechanism-টাই এখানে মূল বিষয়।

নতুন belief:

মানে agent square `(X, Y)` process করা শেষ করেছে।

# 12. Multiple intentions control করা: locking

Locking হলো multiple intentions control করার দ্বিতীয় mechanism। এটি ব্যবহৃত হয় যখন programmer চায় একটি intention-এর short sequence of deeds অন্য intentions-এর interleaving ছাড়া execute হোক।

## 12.1 Mode-based rubble program

Locking section শুরু হয় modes ব্যবহার করে behaviour control করার alternative program দিয়ে:

```gwendolen
+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <- 
        move_to(X, Y);

+!holding(rubble) [achieve] : 
    {B rubble(X, Y), B at(X, Y), B lift_mode} <- 
        lift_rubble;

+at(X, Y) : 
    {~B rubble(X, Y)} <- 
        +no_rubble(X, Y);

+rubble(X, Y) : 
    {B at(X, Y)} <- 
        -search_mode, 
        +lift_mode;

+holding(rubble) : 
    {True} <- 
        print(done);
```

## 12.2 Modes-এর intended meaning

দুইটি mode beliefs আছে:

```gwendolen
search_mode
lift_mode
```

### `search_mode`

Agent বিশ্বাস করলে:

```gwendolen
search_mode
```

first `holding(rubble)` plan applies:

```gwendolen
move_to(X, Y)
```

তাই robot possible rubble locations search করে।

### `lift_mode`

Agent বিশ্বাস করলে:

```gwendolen
lift_mode
```

এবং আরও বিশ্বাস করলে:

```gwendolen
rubble(X, Y)
at(X, Y)
```

second `holding(rubble)` plan applies:

```gwendolen
lift_rubble
```

তাই robot rubble lift করে।

## 12.3 Locking ছাড়া bug

Mode-switch plan:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- 
    -search_mode, 
    +lift_mode;
```

Intended meaning:

1. searching বন্ধ করো;
2. lifting শুরু করো।

কিন্তু intentions interleaved হওয়ায় agent প্রথম deed execute করতে পারে:

```gwendolen
-search_mode
```

তারপর execute করার আগে অন্য intention-এ switch করতে পারে:

```gwendolen
+lift_mode
```

সেই intermediate state-এ belief base-এ নেই:

```gwendolen
search_mode
```

এবং নেই:

```gwendolen
lift_mode
```

এখন original achievement goal reconsidered হয়:

```gwendolen
+!holding(rubble) [achieve]
```

কিন্তু কোনো plan applies না।

First plan requires:

```gwendolen
B search_mode
```

Second plan requires:

```gwendolen
B lift_mode
```

তাই Gwendolen warning/error দেয়:

```text
no applicable plan for goal holding(rubble)
```

Transcript এই intermediate-state problem ব্যাখ্যা করে: `search_mode` remove হয়েছে, কিন্তু `lift_mode` এখনো add হয়নি।

## 12.4 এই problem কেন ঘটে

Two-deed sequence:

```gwendolen
-search_mode, +lift_mode
```

defaultভাবে atomic নয়।

After:

```gwendolen
-search_mode
```

agent অন্য intention select করতে পারে।

Belief base যদি program configuration control করার জন্য ব্যবহৃত হয়, এটি dangerous। Agent half-updated configuration observe করতে পারে।

## 12.5 Locking syntax

Fix হলো mode update lock/unlock deeds দিয়ে ঘিরে দেওয়া:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- 
    +.lock, 
    -search_mode, 
    +lift_mode, 
    -.lock;
```

Slides-এর fixed program:

```gwendolen

    {B possible_rubble(X, Y), ~B no_rubble(X, Y), B search_mode} <- 
        move_to(X, Y);

+!holding(rubble) [achieve] : 
    {B rubble(X, Y), B at(X, Y), B lift_mode} <- 
        lift_rubble;

+at(X, Y) : 
    {~B rubble(X, Y)} <- 
        +no_rubble(X, Y);

+rubble(X, Y) : 
    {B at(X, Y)} <- 
        +.lock, 
        -search_mode, 
        +lift_mode, 
        -.lock;

+holding(rubble) : 
    {True} <- 
        print(done);
```

## 12.6 Lock-এর অর্থ

### মূল ধারণা: lock

**সহজ ধারণা:**  
Lock Gwendolen-কে বলে current intention-কেই select করতে থাকো যতক্ষণ lock release না হয়।

**Lecture-এর formal mechanism:**  
Between:

```gwendolen
+.lock
```

and:

```gwendolen
-.lock
```

current intention remains the current intention। Locked segment চলার সময় অন্য intentions select হয় না।

## 12.7 Worked trace: fixed mode switch

যখন rubble `(5, 5)`-এ perceived হয়, relevant event:

```gwendolen
+rubble(5, 5)
```

Fixed plan expands into deeds:

```gwendolen
+.lock
-search_mode
+lift_mode
-.lock
```

Execution:

1. Execute:

   ```gwendolen
   +.lock
   ```

   Current intention locked হয়।

2. Execute:

   ```gwendolen
   -search_mode
   ```

   Belief `search_mode` removed।

3. Intention locked হওয়ায় Gwendolen এখনো `+!holding(rubble)` intention-এ switch করে না।

4. Execute:

   ```gwendolen
   +lift_mode
   ```

   Belief `lift_mode` added।

5. Execute:

   ```gwendolen
   -.lock
   ```

   Lock released।

6. এখন `holding(rubble)` achievement goal reconsidered হলে lifting plan applies:

   ```gwendolen
   +!holding(rubble) [achieve] : 
       {B rubble(X, Y), B at(X, Y), B lift_mode} <- 
           lift_rubble;
   ```

7. Robot executes:

   ```gwendolen
   lift_rubble
   ```

8. Perception eventually adds:

   ```gwendolen
   holding(rubble)
   ```

9. Final plan prints:

   ```gwendolen
   print(done)
   ```

## 12.8 Locking advice

**Exam flag — lecturer advice:** Lock/unlock নিয়ে সতর্ক থাকতে হবে। Lecturer-এর advice: lock-এর ভিতরে কখনো action রাখবে না, especially এমন action যা execute হতে সময় নেয়।

Reason:

- Locked থাকলে agent অন্য intentions select করতে পারে না।
- Long-running action lock-এর ভিতরে থাকলে agent action চলার সময় অন্য events-এ react করতে পারবে না।
- Lecture example: robot moving করার সময় obstacles check করার intention চলা দরকার হতে পারে। Movement lock-এর ভিতরে থাকলে obstacle-checking intention চলবে না।

### Good use of lock

Short internal configuration updates-এর জন্য lock ব্যবহার করো, যেমন:

```gwendolen
+.lock,
-search_mode,
+lift_mode,
-.lock
```

### Bad use of lock

Avoid:

```gwendolen
+.lock,
move_to(X, Y),
-.lock
```

কারণ `move_to(X, Y)` সময় নিতে পারে এবং moving করার সময় agent responsive থাকা দরকার।

# 13. Waiting vs locking

## 13.1 Waiting

একটি intention কোনো belief true হওয়া পর্যন্ত pause করতে হলে waiting ব্যবহার করো, কিন্তু অন্য intentions চলতে দেওয়া দরকার।

Example:

```gwendolen
move_to(X, Y),
*checked(X, Y)
```

Meaning:

- square-এ move করো;
- square checked হওয়া পর্যন্ত এই intention continue করো না;
- meanwhile, অন্য intentions perceptions process করতে পারে, যেমন `+at(X, Y)` বা `+rubble(X, Y)`।

## 13.2 Locking

একটি short sequence of deeds interleaving ছাড়া execute করা দরকার হলে locking ব্যবহার করো।

Example:

```gwendolen
+.lock,
-search_mode,
+lift_mode,
-.lock
```

Meaning:

- অন্য intention-কে intermediate state observe করতে দিও না, যেখানে neither mode true।

## 13.3 Comparison

```text
Waiting:
    condition true হওয়া পর্যন্ত current intention pause করে;
    অন্য intentions continue করে;
    perceptions/belief updates-এর জন্য অপেক্ষা করতে ভালো।

Locking:
    current intention-কেই current রাখে;
    locked segment চলার সময় অন্য intentions run করে না;
    short atomic belief/configuration updates-এর জন্য ভালো।
```

**Exam flag — contrast:** Waiting responsiveness preserve করে; locking সাময়িকভাবে responsiveness কমায়।

## 14.2 Baseline rubble program with extra move

Actions:

```text
move_to(1, 1)
move_to(3, 3)
move_to(5, 5)
lift_rubble
move_to(5, 5)
print(done)
```

Extra move-এর কারণ:

```text
lift_rubble holding(rubble)-এর perception তৈরি করে
কিন্তু holding(rubble) এখনো belief base-এ নেই
তাই achieve goal আরেকবার reconsidered হয়
এবং move_to(5, 5) আবার selected হয়
```

## 14.3 Reasoning-rule rubble program

Reasoning rule:

```gwendolen
square_to_check(X, Y) :- 
    possible_rubble(X, Y), 
    ~no_rubble(X, Y);
```

Plan:

```gwendolen
+!holding(rubble) [achieve] : {B square_to_check(X, Y)} <- 
    move_to(X, Y);
```

Behaviour:

```text
Baseline program-এর মতো একই behaviour,
কিন্তু guard logic reasoning rule-এ factored করা হয়েছে।
```

## 14.4 List-recursive rubble program

Initial belief:

```gwendolen
possible_rubble([sq(1, 1), sq(3, 3), sq(5, 5)])
```

Rules:

```gwendolen
square_to_check(X, Y) :- 
    possible_rubble(L), 
    check_rubble(L, X, Y);

check_rubble([sq(X, Y) | T], X, Y) :- 
    ~no_rubble(X, Y);

check_rubble([sq(X, Y) | T], X1, Y1) :- 
    no_rubble(X, Y), 
    check_rubble(T, X1, Y1);
```

Behaviour:

```text
List-এর first unchecked square return করে।
After no_rubble(1, 1), recurse to sq(3, 3)।
After no_rubble(3, 3), recurse to sq(5, 5)।
```

## 14.5 Goal-in-guard rubble relocation

Initial goal:

```gwendolen
rubble(2, 2) [achieve]
```

Plan:

```gwendolen
+!rubble(2, 2) [achieve] : {True} <- 
    +!holding(rubble) [achieve], 
    move_to(2, 2), 
    drop;
```

Guarded lifting plan:

```gwendolen
+rubble(X, Y) : 
    {B at(X, Y), G holding(rubble) [achieve]} <- 
        lift_rubble;
```

Behaviour:

```text
Rubble খুঁজে বের করে।
Only if currently pursuing holding(rubble), তখনই lift করে।
Move to (2, 2)।
Drop করে।
```

## 14.6 Waiting rubble program

Key plan:

```gwendolen
+!holding(rubble) [achieve] : 
    {B possible_rubble(X, Y), ~B no_rubble(X, Y)} <- 
        move_to(X, Y),
        *checked(X, Y);
```

Effect:

```text
Square-এ move করার পরে checked(X, Y) believed হওয়া পর্যন্ত wait করে।
এটি premature repeated movement ঠেকায়।
```

## 14.7 Locking mode-switch program

Buggy mode-switch:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- 
    -search_mode, 
    +lift_mode;
```

Bug:

```text
Agent -search_mode-এর পরে কিন্তু +lift_mode-এর আগে switch করতে পারে।
তখন search plan বা lift plan — কোনোটাই apply করে না।
```

Fixed mode-switch:

```gwendolen
+rubble(X, Y) : {B at(X, Y)} <- 
    +.lock, 
    -search_mode, 
    +lift_mode, 
    -.lock;
```

Effect:

```text
অন্য intention run করার আগে mode update complete হয়।
```

# 15. Exam flags and high-value points

## 15.1 Plans are event-triggered

**Know this:** কোনো belief বা goal add/remove হলে plan triggered হয়; কোনো condition true থাকলেই triggered হয় না।

Obstacle example:

```gwendolen
+obstacle : {T} <- turn_90, forward.
```

Robot বারবার turn করে না কারণ `obstacle` true-ই থাকে, আবার appear করে না।

## 15.2 Guards express current conditions

Triggers events-এর জন্য। Guards currently hold করা conditions-এর জন্য।

Example:

```gwendolen
+at(X, Y) : {~B rubble(X, Y)} <- +no_rubble(X, Y);
```

Event:

```gwendolen
+at(X, Y)
```

Condition:

```gwendolen
~B rubble(X, Y)
```

## 15.3 Achieve vs perform goals

Perform goals:

```text
plan, execute, drop
```

Achieve goals:

```text
belief base check;
not achieved হলে plan and execute;
again check;
still not achieved হলে plan again
```

## 15.4 Intentions are stacks of tuples

Tuple structure জানো:

```text
(trigger event, deed, unifier)
```

Special deeds জানো:

```text
no plan yet
null
```

## 15.5 `no plan yet` মানে planning এখনো হয়নি

Top deed যদি:

```text
no plan yet
```

হয়, Gwendolen applicable plans search করে।

## 15.6 Perception creates intentions

Perceived information instantly belief নয়। এটি belief add করার intention তৈরি করে।

এটাই কারণ rubble program rubble lift করলেও agent এখনো বিশ্বাস নাও করতে পারে:

```gwendolen
holding(rubble)
```

## 15.7 Extra move explainable

Extra:

```gwendolen
move_to(5, 5)
```

ঘটে কারণ `holding(rubble)` belief add হওয়ার আগে achieve-goal intention আবার selected হয়।

## 15.8 Reasoning rules guards simplify করতে পারে

সব কিছু guard-এ না লিখে:

```gwendolen
{B possible_rubble(X, Y), ~B no_rubble(X, Y)}
```

define:

```gwendolen
square_to_check(X, Y) :- possible_rubble(X, Y), ~no_rubble(X, Y);
```

and use:

```gwendolen
{B square_to_check(X, Y)}
```

## 15.9 Lists and recursion are allowed in reasoning rules

List version uses:

```gwendolen
[sq(X, Y) | T]
```

and recursive `check_rubble`।

## 15.10 Goals can appear in guards

Example:

```gwendolen
G holding(rubble) [achieve]
```

এটি check করে agent currently ওই goal রাখে কি না।

## 15.11 Waiting controls progress of one intention

Wait syntax:

```gwendolen
*checked(X, Y)
```

Meaning:

```text
checked(X, Y) believed না হওয়া পর্যন্ত এই intention continue করো না।
```

## 15.12 Locking prevents interleaving

Lock syntax:

```gwendolen
+.lock
...
-.lock
```

Meaning:

```text
Lock release না হওয়া পর্যন্ত current intention-কেই current রাখো।
```

## 15.13 Locking warning

**Lecturer advice:** Long-running actions lock-এর ভিতরে রেখো না। বিশেষ করে movement-এর মতো action lock-এর ভিতরে রাখো না যদি robot-কে obstacles বা অন্য events-এ responsive থাকতে হয়।
